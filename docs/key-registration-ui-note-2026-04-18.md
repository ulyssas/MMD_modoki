# キー登録 UI 配置メモ

更新日: 2026-04-18

関連:
- [左パネル UI 案メモ](./left-panel-ui-ideas-2026-04-18.md)
- [MMD キーフレーム仕様の調査メモ](./mmd-keyframe-features-survey.md)
- [キーフレ登録表示の調査メモ](./keyframe-registration-display-research.md)

## 1. 目的

キー登録まわりの UI をどこへ置くかを整理する。

特に次の点を明確にしたい。

- 左パネルにキー登録を置かない場合、どこへ置くか
- 下パネルと viewport 直下で何を分担するか
- 「現在値の登録」と「既存キーの操作」をどう分けるか

## 2. 前提

左パネルは時間軸ナビゲーションに寄せたい。

そのため左パネルでは、

- 現在フレーム
- Start / Stop
- Loop
- 前後フレーム移動
- 前後キー移動
- タイムライン
- 波形表示

を優先し、キー登録ボタン群はできるだけ置かない方針とする。

## 3. 難しさ

キー登録まわりは、見た目以上に役割が混ざりやすい。

混ざりやすい要素:

1. 現在値をキーとして登録する操作
2. 既存キーを選択して操作する機能
3. 複数キーをまとめて扱う機能
4. ボーン / モーフ / カメラで対象が違うこと

このため、単純に「登録ボタンをどこへ置くか」だけでは整理しきれない。

## 4. 操作の分解

最低限、次の 3 系統に分けて考える。

### 4-1. 現在値を登録する

例:

- 現在のボーン位置 / 回転を登録
- 現在のモーフ値を登録
- 現在のカメラ値を登録

これは「いま見えている値」をキー化する操作なので、値編集に近い。

### 4-2. 既存キーを操作する

例:

- 選択キーを削除
- 選択キーをコピー
- 選択キーを反転
- 選択キーの補間を編集

これは時間軸上で既に存在するキーを対象にする操作。

### 4-3. 複数キーをまとめて操作する

例:

- 矩形選択したキーをまとめて移動
- 範囲選択したキーを削除
- 複数キーを複製

これはタイムライン選択と強く結びつく。

## 5. 配置候補

### 5-1. 左パネル

長所:

- タイムラインに近い
- 時間軸操作と一緒に見られる

短所:

- 左パネルを時間軸ナビゲーションに絞りにくくなる
- ボタン群を足し始めるとすぐ混雑する
- 現在値登録と既存キー操作が混ざりやすい

結論:

- 左パネルは本命ではない
- 既存キー操作の入口を少し置く余地はあるが、登録の主置き場にはしない

### 5-2. 下パネル

長所:

- 値編集と近い
- 対象別 UI と並べやすい
- 「現在値を登録する」操作と相性が良い

短所:

- 既存の値編集 UI が多く、詰め込み過ぎると密度が高くなる
- タイムライン選択と少し距離がある

結論:

- キー登録の主置き場として本命
- 登録 / 削除 / 補間 / コピー系をまとめやすい

### 5-3. viewport 直下

長所:

- gizmo と近い
- 操作値の変化を見ながら登録しやすい
- 本家 MMD で viewport に重なっていた簡易コントローラの受け皿になりうる

短所:

- まだ設計段階であり、UI 面の責務が固まっていない
- ボーン / カメラ / アクセサリで共通設計が必要
- キー編集アクションまで入れると重くなりやすい

結論:

- 現在値操作面としては有望
- ただしキー登録 UI の主置き場にするかは未確定

## 6. 現時点の分担案

### 左パネル

- 時間軸の確認
- 移動
- 範囲指定
- タイムライン選択

### 下パネル

- 現在値の登録
- 選択キーの削除
- 補間
- コピー / ペースト / 反転
- 対象別の詳細編集

### viewport 直下

- gizmo と連動する現在値表示
- 座標系 / モード表示
- transform / camera 値の直接編集
- 必要なら軽い `登録` ボタン

この分担なら、

- 左 = どこを触るか
- 下 = 何をキー化 / 編集するか
- viewport 直下 = いまの値をどう動かすか

で整理しやすい。

## 7. 本命案

当面の本命は次の形。

1. 左パネルからキー登録 UI は外す
2. 下パネルにキー登録関連を寄せる
3. viewport 直下は別件として、現在値操作面として設計する

理由:

- 左パネルをすっきり保てる
- 既存値編集との関係が分かりやすい
- 後から viewport 直下 UI を足しても衝突しにくい

## 8. viewport 直下に置きたい要素

viewport 直下は `現在値操作面` として使う想定がよい。

相性が良い要素:

- Local / Global 軸切替
- gizmo モード切替
- gizmo と連動する位置 / 角度の数値表示
- 数値直接入力
- 選択対象の現在値確認

`登録` ボタンも置く余地はあるが、主役にはしない方がよい。

整理:

- viewport 直下 = 現在値を触る場所
- 下パネル = キー編集の主導線

この前提なら、viewport 直下の `登録` は補助導線として扱うのがよい。

## 9. オートキー登録

オートキー登録は、viewport 直下 UI と相性が良い。

想定フロー:

1. gizmo を動かす
2. 数値が変わる
3. 必要なら現在フレームへ自動でキーを打つ

ただし、雑に入れると事故りやすい。

先に決めるべきこと:

- 対象
  - ボーンだけか
  - カメラも含むか
  - モーフも含むか
- 発火条件
  - gizmo 操作終了時
  - 数値入力確定時
  - スライダ変更時
- 既存キーがない場合の扱い
- 補間の初期値

当面の本命案:

- 対象はボーンだけ
- 発火は gizmo 操作終了時と数値入力確定時だけ
- カメラは手動登録を維持する
- モーフの連続スライダ変更にはまだ使わない
- viewport 直下に `Auto Key` トグルを置く
- 実装は新経路を増やさず、既存の「現在値登録」を内部で呼ぶ

つまり、オートキーは単独機能というより、viewport 直下 UI を作るなら一緒に設計すべき機能と考える。

補足:

- ボーンは gizmo 操作との相性が良く、`動かしたら打つ` が自然
- カメラは試行錯誤の途中で値を触ることが多く、自動登録だとキーが増えすぎやすい
- そのため、カメラは `現在値確認 -> 明示登録` の流れを維持する方がよい

## 10. 実装順メモ

### Step 1

- 左パネルを時間軸専用へ寄せる
- 下パネルでキー登録 / 削除の位置を整理する

### Step 2

- 補間 / コピー / 反転の配置を下パネル側で揃える

### Step 3

- viewport 直下 UI の設計メモを別途切る
- その上で、現在値操作と登録ボタンの距離を再検討する

### Step 4

- viewport 直下 UI とオートキー登録の関係を整理する
- `Auto Key` を入れるなら対象と発火条件を限定して導入する

## 11. 現状の MMD_modoki に足りないキー登録系機能

本家 MMD の画面を見ると、MMD_modoki でまだ薄い機能は「登録ボタンそのもの」より、その前後の補助操作に多い。

### 11-1. 現フレームに対するキー操作

不足候補:

- 現フレームキーのコピー
- 現フレームキーのペースト
- 反転ペースト
- 現フレームキーの削除

これは登録後の微調整や、左右対称作業の効率に直結する。

### 11-2. 範囲 / 複数キーに対する操作

不足候補:

- 範囲選択
- 矩形選択
- 範囲選択後の削除
- 範囲選択後のコピー / 複製
- 複数キーのまとめ移動

本家 MMD では左側に強い導線があるが、MMD_modoki ではタイムライン選択モデルと一緒に整理が必要。

### 11-3. 複数ボーン登録を助ける選択補助

不足候補:

- BOX選択
- 全て選択
- 未登録選択

これは単なる選択機能ではなく、複数ボーンを一括登録するための前段操作として重要。

### 11-4. 補間まわりの補助操作

不足候補:

- 補間のコピー
- 補間のペースト
- 自動設定
- 線形補間への一括切り替え

MMD_modoki は補間 UI 自体の整理も残っているため、登録 UI と切り離さずに考える必要がある。

### 11-5. 対象別の登録導線

不足候補:

- ボーンの登録 / 初期化
- モーフの登録 / リセット
- カメラの登録 / 初期化

本家 MMD は対象ごとに「いまの値を登録する」導線がかなり近い。MMD_modoki でも下パネル側で対象別に揃える余地がある。

### 11-6. 物理焼き込みキー

不足候補:

- 物理結果をボーンキーへ焼き込む
- 指定範囲をまとめて焼き込む
- 焼き込み前後のプレビュー確認
- 焼き込み対象の限定

これは通常の「現フレームを登録する」機能とは別系統で、重量級の機能として扱うべき。

重い理由:

- 単フレーム登録ではなく範囲処理になる
- 物理シミュレーションの安定性に依存する
- ボーンキーの大量生成と整合確認が必要
- 焼き込み後の編集導線も必要になる

そのため、下パネルに単純な登録ボタンを増やす話とは分けて設計した方がよい。

### 11-7. 現状の扱い

現状の MMD_modoki は、VMD 由来の物理焼き込みキー相当を完全には無視していない。

現状整理:

- VMD 読み込み自体は `babylon-mmd` の `VmdLoader` に渡している
- `babylon-mmd` 側は VMD の物理トグル情報を `physicsToggles` として読み取る
- runtime 再生でも `physicsToggles` は参照される
- MMD_modoki 側でも、VMD マージや project 保存 / 読み込みでは `physicsToggles` を保持している

ただし、MMD_modoki 自身はこれを第一級の編集対象としては扱っていない。

未対応寄りの点:

- 物理焼き込みキーの存在を UI 上で表示していない
- 物理焼き込みキーを個別に編集する導線がない
- 手動キー登録時は `physicsToggles` を近傍キーから引き継ぐだけ

つまり現状は、

- VMD 由来の物理焼き込みキーは pass-through 的に保持される
- ただし MMD_modoki 側では未対応寄り

という扱いに近い。

### 11-8. 今後の優先順

優先度の高い順:

1. 現フレームキーの削除 / コピー / ペースト
2. 範囲選択と複数キー操作
3. 複数ボーン登録のための選択補助
4. 補間補助操作
5. 物理焼き込みキーは別タスクとして重めに設計する

## 12. 保留事項

- ボーン、モーフ、カメラで完全共通の登録 UI にするか
- 複数キー選択時の登録 / 削除ボタンの意味をどうするか
- viewport 直下に登録ボタンを置くか、値だけ置くか
- 本家 MMD の重複した導線をどこまで整理して削るか

## 13. 現時点の結論

キー登録は左パネルではなく、下パネル側へ寄せる前提で考えるのが一番整理しやすい。

viewport 直下 UI は有望だが、まずは「現在値操作面」として分けて扱い、キー登録 UI の主置き場とは切り離して考える。
