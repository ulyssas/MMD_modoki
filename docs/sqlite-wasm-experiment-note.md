# SQLite WASM 実験メモ

## 概要

このメモは、`MMD_modoki` に `SQLite WASM` を導入する価値があるかを整理するための実験メモである。

現時点での前提は以下。

- これは `本番基盤の即採用案` ではなく、`実験的な研究テーマ`
- 主目的は `ログ管理`、`入力イベント管理`、`設定/プロファイル管理` の一元化
- `undo/redo` の中核保存先としてすぐ採用する前提ではない
- このアプリは当面 `技術的試作 / 実験機` として扱うため、こうした基盤実験自体には意味がある

## 背景

現在のアプリでは、将来的に次のような入力や状態が増える可能性がある。

- キーボードショートカット
- マウス / ギズモ操作
- タイムライン編集
- `MIDI` コントローラー
- `Gamepad`
- 将来の外部入力デバイス

これらが増えると、`何がいつ発生し、どの設定で、何にマッピングされ、最終的に何を動かしたか` を追うのが難しくなる。

そのため、単なるログ機能ではなく、`入力イベント基盤 + 観測基盤 + 設定基盤` をある程度まとめて扱える構造が欲しくなる。

## 期待していること

`SQLite WASM` に期待しているのは主に次の点。

- ログ、入力イベント、設定変更履歴を同じモデルで扱えること
- `MIDI / Gamepad / ショートカット` の入力イベントを同じテーブル設計で記録できること
- あとから `いつ何が起きたか` を検索しやすいこと
- 入力マッピングやデバイスプロファイルを構造化して保存できること
- UI 側で簡易ビューアやフィルタを作りやすいこと
- 将来、操作解析や簡易的なマクロ研究に流用しやすいこと

## 向いていそうな用途

最初に試すなら、次の用途が向いている。

### 1. イベントログ

- `input_event`
- `command_event`
- `error_event`
- `render_event`

のような形で、イベント系列を一元記録する。

### 2. 入力デバイス設定

- デバイス一覧
- デバイスごとの有効/無効
- 入力マッピング
- プリセット保存

を構造化して扱う。

### 3. 設定変更履歴

- 設定値の変更
- 実験機能の ON/OFF
- ユーザー環境差分

を記録し、調査しやすくする。

## まだ向いていない用途

現時点では、次の用途にすぐ広げるのは重い。

### 1. undo/redo の本体

`undo/redo` は保存先より、`操作の粒度`、`逆操作の設計`、`副作用の分離` が先に必要。
そのため、先に `SQLite WASM` を入れても決定打にはなりにくい。

### 2. プロジェクト保存の主形式

プロジェクト本体をいきなり DB 化すると、可搬性、差分確認、手動救済が難しくなる。
今は `JSON 系の保存形式` のほうが実務上扱いやすい。

### 3. コアデータの全面的な RDB 化

scene、timeline、material、physics まで一気に寄せるのは過剰。
アプリ全体の責務整理が先。

## SQLite 公式ドキュメントとの対応

`SQLite WASM` で `undo/redo` や `in-memory RDB` を考えるとき、完全にゼロからの発想というわけではない。
SQLite 公式には、そのまま、あるいは発想元として使える資料がある。

### 1. In-Memory Databases

- 公式: `https://www.sqlite.org/inmemorydb.html`
- `:memory:` や一時 DB の扱いを説明している
- `in-memory RDB` として試すときの基本前提はここ

### 2. Automatic Undo/Redo Using SQLite

- 公式: `https://www.sqlite.org/undoredo.html`
- `UNDOLOG` テーブルと `INSERT/UPDATE/DELETE` トリガーで undo/redo を組む設計ノート
- アプリの変更を SQL として逆操作できるように積む考え方
- `MMD_modoki` でそのまま採用するには重いが、`SQLite を操作履歴に使うとしたらどう考えるか` の参考としてかなり重要

### 3. Session Extension

- 公式: `https://www.sqlite.org/sessionintro.html`
- `changeset` / `patchset` を使って変更差分を取る拡張
- `undo/redo` 専用というより、差分記録、比較、適用、巻き戻し検討の土台として近い

### 現時点での見方

- 公式に `SQLite WASM の in-memory DB でアプリ undo/redo をこう実装せよ` という専用ガイドがあるわけではない
- ただし、`undo/redo` を SQLite で扱う発想自体は公式資料がある
- 実験するなら、
  - `UNDOLOG + trigger` 方式
  - `Session Extension` 的な差分方式
  のどちらがこのアプリに合うかを比較する価値がある

## SQLite WASM を使う利点

- `in-memory RDB` として扱える
- 検索、集計、絞り込みがしやすい
- ログ閲覧 UI や入力解析 UI に流用しやすい
- 実験的機能の観測基盤として面白い
- `技術的試作` という位置づけと相性がよい

## SQLite WASM を使う懸念

- 現時点では明らかに過剰になりうる
- Electron アプリで `WASM + 永続化` を持つぶん複雑さが増える
- バイナリ DB は、素のログファイルより直接読みにくい
- `DB を入れれば整理される` わけではなく、イベントモデル設計が先
- 調査用途では `text/jsonl` のほうが即効性が高い場面も多い

## 実験で確認したい事項

`SQLite WASM` を本当に試すなら、便利そうかどうかだけでなく、速度と運用負荷を確認する必要がある。

### 性能面

- 高頻度の `input_event` を流しても UI が重くならないか
- renderer 側で記録してもフレーム落ちが出ないか
- バッチ insert と逐次 insert のどちらが現実的か
- ログ件数が増えたときに検索速度がどの程度維持されるか
- メモリ消費が許容範囲に収まるか

### 永続化面

- 永続化あり / なしでどの程度体感差があるか
- セッション終了時 flush のコストが大きすぎないか
- ログ破損時の復旧や切り戻しがどれだけ面倒か

### 運用面

- `text/jsonl` より本当に調査しやすいか
- ログビューアや簡易検索 UI を作る価値があるか
- `MIDI` / `Gamepad` / ショートカット設定の一元管理先として自然か
- 観測基盤として便利でも、アプリ全体を複雑化しすぎないか

### 結論判定の観点

- `便利そう` ではなく `実用速度が出るか`
- `実験として面白い` だけでなく `保守コストに見合うか`
- 既存の `テキストログ + JSON 設定` より明確な利点があるか

## 導入するならの方針

導入する場合は、最初からアプリ中核には置かない。

### Phase 1

- 完全に実験用の独立レイヤとして置く
- `input_event` と `error_event` を記録するだけに絞る
- 既存ログファイルも残す

### Phase 2

- デバイスプロファイル保存を試す
- `MIDI` / `Gamepad` / ショートカット設定を一部 DB へ寄せる

### Phase 3

- 設定変更履歴やイベント検索 UI を試す
- 実際に運用メリットがあるか評価する

### Phase 4

- 必要なら `undo/redo` 周辺との接続可能性を再検討する
- ここで初めて中核採用の是非を判断する

## 現時点の結論

`SQLite WASM` は、今のアプリに必須ではない。
ただし、`実験機としてのこのリポジトリ` にはかなり相性がよい。

なので結論は次の通り。

- `本筋の必須基盤` として急いで入れるものではない
- ただし `実験的な観測基盤・入力基盤` としては試す価値がある
- 最初は `ログ / 入力イベント / 設定プロファイル` の限定用途に絞る
- `undo/redo` やプロジェクト保存の中核に据えるのはまだ早い
