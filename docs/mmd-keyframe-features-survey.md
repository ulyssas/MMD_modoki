# 本家MMDのキーフレーム関連機能調査

## 概要

本資料は、`MikuMikuDance` 本体が持つキーフレーム関連機能を洗い出すための調査メモです。  
`MMD_modoki` の機能比較や、将来の互換性検討のための参照用としてまとめています。

調査日:

- 2026-03-15

調査方針:

- 画面上の機能は、MMD 操作マニュアル系の公開資料と VPVP wiki の更新履歴で確認
- 保存できるキーフレーム種別は、VMD/VPD の仕様整理資料で確認
- 明確に確認できた事実と、複数資料からの推定は分けて扱う

注意:

- MMD にはまとまった公式仕様書が見当たらず、公開 wiki や解説資料を突き合わせて整理しています
- UI の基本構成は古いマニュアル資料でも確認できますが、現行 9.x 系までの細かな差分は更新履歴で補っています
- 本資料では「本家MMDで標準搭載されている機能」を対象とし、MMM やプラグイン独自機能は原則含めていません

## 結論の要約

本家MMDのキーフレーム機能は、大きく分けると次の 4 系統です。

1. キーフレームの対象そのもの
   ボーン、表情、カメラ、照明、表示/IK、セルフシャドウ、アクセサリ
2. タイムライン編集
   登録、削除、コピー、ペースト、反転ペースト、範囲選択、拡大縮小、全選択など
3. 補間制御
   補間グラフ編集、補間のコピー/ペースト、線形補間、自動設定
4. 補助的な一括処理
   不要フレーム削除、空フレーム挿入、列フレーム削除、各種バイアス付加、補正系

`MMD_modoki` で互換性を意識する場合、優先度が高いのは次です。

- ボーン、モーフ、カメラ、表示/IK のキーフレーム
- タイムライン上のコピー/ペースト/削除/範囲選択
- 補間グラフ編集
- フレーム挿入/削除と時間方向の調整

## MMD_modoki での再現案

この節では、本家MMDのキーフレーム機能に対して、`MMD_modoki` の現在地と再現方針を整理します。

評価の見方:

- `実装済み`: すでにアプリ上で概ね使える
- `部分対応`: 中核はあるが、本家相当には不足がある
- `未対応`: 現状の実装範囲では見当たらない

### 1. 対象カテゴリごとの再現案

| 本家MMDの対象 | MMD_modoki 現状 | 再現案 |
| --- | --- | --- |
| ボーンキーフレーム | `実装済み` | `root / semi-standard / bone` の track を持ち、キー追加・削除・移動・補間編集まで既にある。この系統を基準に機能を広げる |
| モーフキーフレーム | `部分対応` | モーフのキー自体は扱えるが、補間編集は無効。まずは本家同様に `値の登録・削除・コピー/ペースト` を優先し、補間は後回しでよい |
| カメラキーフレーム | `部分対応` | キー追加・削除・移動・補間編集はある。今後は `コピー/ペースト`、`別フレーム貼り付け`、`範囲編集` を追加すると本家に近づく |
| 照明キーフレーム | `未対応` | 既存の light state はあるので、`LightingTrack` を 1 本追加し、色と方向を project と timeline に載せれば再現可能 |
| 表示 / IK キーフレーム | `未対応` | project serializer には property track 構造があるため、UI と timeline track を追加して `表示` と `IK ON/OFF` を編集可能にするのが最短 |
| セルフシャドウキーフレーム | `未対応` | effect / lighting state の一部として 1 トラック化する案が現実的。優先度は低い |
| アクセサリキーフレーム | `未対応` | すでに accessory 自体は拡張で存在する。transform / visible / parent を時間軸に載せれば本家に近い編集が可能 |

| 本家MMDの機能 | MMD_modoki 現状 | 再現案 |
| --- | --- | --- |

### 3. 補間機能ごとの再現案

| 本家MMDの機能 | MMD_modoki 現状 | 再現案 |
| --- | --- | --- |
| ボーン補間グラフ編集 | `実装済み` | 現在の interpolation panel を維持しつつ、copy / paste を追加する |
| カメラ補間グラフ編集 | `実装済み` | 既存機能あり。UI 上でチャンネルコピーを足せば本家にかなり近い |
| モーフ補間編集 | `未対応` | 本家MMDでもモーフは単純値運用が中心なので、優先度は低い。必要なら専用仕様を後付けする |
| 線形補間 | `部分対応` | 現在は curve の直接編集が中心。`全 control point を線形値に戻す` ボタンを加えると再現しやすい |
| 補間コピー / ペースト | `未対応` | `InterpolationCurve` の snapshot を clipboard 化して track 間コピーできるようにする |
| 補間自動設定 | `未対応` | 最初は簡易版で十分。新規キー追加時に前後キーから標準ベジェを自動生成する |

### 4. 実装順のおすすめ

本家MMDに寄せる目的で優先度を付けるなら次の順番が妥当です。

1. `コピー / ペースト / 別フレームペースト`
2. `範囲選択` と `複数キー移動 / 削除`
3. `表示 / IK キーフレーム`
4. `空フレーム挿入 / 列フレーム削除 / 拡大縮小`
5. `反転ペースト`
6. `照明キーフレーム`
7. `アクセサリキーフレーム`
8. `セルフシャドウキーフレーム`

この順番を勧める理由:

- 1 から 4 までは既存 timeline 実装の延長で乗せやすい
- 5 は左右対応表や変換規則が要るので一段重い
- 6 から 8 は timeline の対象カテゴリ自体を増やす必要があり、設計変更の影響が大きい

### 5. 実装の土台として既に使えるもの

`MMD_modoki` には、再現に使える土台がすでにいくつかあります。

- timeline track category と keyframe frame map
- キー追加 / 削除 / 移動 API
- camera keyframe の編集準備処理
- ボーン / カメラの補間カーブ編集 UI
- project export / import 時の animation serialize
- property track を扱う内部データ構造

つまり、完全にゼロから作る必要があるのは少なく、足りないのは主に次です。

- 複数選択モデル
- clipboard 操作
- property / lighting / accessory 用の timeline category
- 一括編集コマンド群

## Babylon.js Editor / babylon-mmd が提供している機能

この節では、`MMD_modoki` 側で新規実装しなくても流用または参考にできる機能が、`Babylon.js Editor` や `babylon-mmd` にあるかを整理します。

### 1. Babylon.js Editor について

#### 確認できたこと

- `Babylon.js Editor` は Babylon.js 用のビジュアル scene editor で、scene、material、asset、script の編集を主目的としている
- 公式サイトでは、`animated 3D models with skeletons support` をうたっている
- 公式サイトでは、v5 の cinematic editor の proof of concept により、短編映像を `scene assembly and lighting to animating` まで Editor 内で作ったと説明している
- 公式ドキュメントで明示的に確認できた animation 編集機能は、少なくとも `Sprite Manager` 向けの animation 作成、再生、削除

#### 調査上の判断

- `Babylon.js Editor` は `MMD_modoki` に組み込むための軽量ライブラリというより、別アプリの editor 製品として考えるべき
- 公式ドキュメント上では、MMD のような `ボーン / カメラ / モーフ / 表示IK` を同一 timeline で編集する汎用キーフレーム editor は確認できなかった
- よって `Babylon.js Editor` をそのまま使って `MMD_modoki` のキーフレーム editor を肩代わりさせるのは現実的ではない

#### MMD_modoki での活用可能性

- `参考になるもの`
  - scene editor としての UI 設計
  - Babylon.js 上での animation 付き 3D asset の扱い方
  - 将来的な cinematic workflow の参考
- `期待しにくいもの`
  - MMD 専用 timeline editor
  - VMD 互換のキーフレーム編集
  - in-app component としてそのまま埋め込める editor 機能

#### 結論

`Babylon.js Editor` は `MMD_modoki` のキーフレーム機能を直接提供するものではなく、主に `設計参考` の位置づけです。  
「本家MMD互換の editor 機能を外部ライブラリで丸ごと賄う」候補にはなりにくいです。

### 2. babylon-mmd について

#### 確認できたこと

- `babylon-mmd` は Babylon.js 向けの MMD loader / runtime であり、PMX/PMD、VMD/VPD、physics、IK、morph、audio sync を提供している
- README の implementation status では、VMD/VPD loader、MMD morph system、IK、audio sync、animation blending、basic animation player UI が確認できる
- `Use Babylon.js Animation Runtime` のドキュメントでは、MMD animation を Babylon.js の `Animation` / `AnimationGroup` 系へ変換して使う方法が説明されている
- 同ドキュメントでは、その方法の利点として `Babylon.js Animation Curve Editor support` が明示されている

#### ただし制約もある

- Babylon.js 標準 animation system は MMD のベジェ補間をそのまま表現できない
- そのため `babylon-mmd` 側で Hermite 近似、sample 近似、独自ベジェ実装の 3 方式を用意している
- sample 方式は精度が高い一方で `animations become non-editable`
- 独自ベジェ実装方式は精度が高い一方で、`Animation Curve Editor may not work properly`
- README では `Self shadow, Ground shadow spec support` は `Not planned features` に入っている
- built-in の player control は legacy docs で `testing purposes only` かつ `customization は多くない` とされている

#### MMD_modoki での活用可能性

- `そのまま活用できる部分`
  - PMX/PMD/VMD/VPD の parse / load
  - MMD runtime
  - morph / IK / physics / audio sync
  - camera / model animation の bind
  - animation blending の土台
- `部分的に活用できる部分`
  - Babylon.js `Animation` / `AnimationGroup` への変換
  - curve editor 連携
    - ただし builder の選択次第で editability や正確性に制約がある
- `別途実装が必要な部分`
  - MMD 本家風の timeline UI
  - コピー / ペースト / 範囲選択 / 反転ペースト
  - 表示 / IK / ライト / アクセサリの editor UX
  - project editor としての一括編集コマンド

#### 結論

`babylon-mmd` は `MMD_modoki` の下位基盤としてかなり有用です。  
ただし提供しているのは主に `loader / runtime / 再生基盤` であり、`本家MMD相当のキーフレーム editor` までは提供していません。

### 3. 総合判断

結論を短くまとめると次の通りです。

- `Babylon.js Editor`
  - 直接の機能提供元というより参考実装
  - `MMD_modoki` の editor 機能を肩代わりする候補にはなりにくい
- `babylon-mmd`
  - 直接の機能提供元として有用
  - loader / runtime / animation 変換の土台はかなり任せられる
  - ただし editor UX と timeline 編集機能は `MMD_modoki` 側で作る必要がある

したがって、`MMD_modoki` の方針としては次が現実的です。

1. `babylon-mmd` を runtime / animation data 基盤として最大限活用する
2. keyframe editor、clipboard、複数選択、範囲編集などの UX は `MMD_modoki` 側で実装する
3. `Babylon.js Editor` は機能移植先ではなく、UI や cinematic workflow の参考として見る

## 1. キーフレーム対象の種類

### 1.1 ボーンキーフレーム

確認できた内容:

- ボーンごとにキーフレームを登録できる
- ボーン操作パネルに `フレーム登録` がある
- VMD 上では、ボーン名、フレーム番号、位置、回転、補間、物理トグルが保存される

確認ソース:

- `パネル・ボタン一覧` にボーン操作パネルの `フレーム登録` がある
- `VMDとVPDの概要` にボーンキーフレームの保存内容がある

補足:

- ボーンの補間は位置 `X/Y/Z` と回転ごとに持つ
- 物理ボーン名の長さ制限により、VMD 再読込時に bind 失敗が起こることがある

### 1.2 モーフキーフレーム

確認できた内容:

- 表情操作パネルからモーフ値を登録できる
- VMD 上では、モーフ名、フレーム番号、値が保存される

確認ソース:

- `パネル・ボタン一覧` に表情操作パネルの `登録` がある
- `VMDとVPDの概要` にモーフキーフレームの保存内容がある

補足:

- 値は `0.0` から `1.0`
- VPD にモーフ状態を入れるのは MMM 拡張であり、本家MMDの標準仕様ではない

### 1.3 カメラキーフレーム

確認できた内容:

- カメラ位置、角度、距離、視野角をキーフレームとして登録できる
- カメラ操作パネルに `登録` がある
- VMD 上では、フレーム番号、距離、中心位置、回転、補間、視野角、投影タイプが保存される

確認ソース:

- `カメラを動かしてみよう`
- `パネル・ボタン一覧`
- `VMDとVPDの概要`

補足:

- MMD のカメラは 1 台のみ
- カメラはオービット型で、`カメラの位置` というより `中心位置 + 距離 + 回転` の構造
- 複数カメラを持つ仕組みはなく、カット切替は単一カメラのキーフレームで作る

### 1.4 照明キーフレーム

確認できた内容:

- 照明色 `RGB` と向き `X/Y/Z` をキーフレーム登録できる
- 照明操作パネルに `登録` がある
- VMD 上では、ライトの方向と色が保存される

確認ソース:

- `パネル・ボタン一覧`
- `VMDとVPDの概要`

補足:

- MMD は 1 つの directional light 前提
- ライトキーフレームは存在するが、利用頻度は高くないとされる

### 1.5 表示 / IK プロパティキーフレーム

確認できた内容:

- モデルの `表示` 状態をキーフレームで切り替えられる
- 各 IK ボーンの `ON/OFF` をキーフレームで切り替えられる
- これらはフレーム操作パネル上の `表示・IK・外観` 系の行に登録される
- VMD 上では、可視状態と IK 有効状態テーブルが保存される

確認ソース:

- `モデル操作パネル ～ 表示・セルフ影・加算・IKボーン の ON/OFF ～`
- `VMDとVPDの概要`

補足:

- `表示` と `IK ON/OFF` は、ボーンやモーフとは別カテゴリのキーフレームとして扱われる
- モデル操作パネルの `登録` ボタン経由で記録される

### 1.6 セルフシャドウキーフレーム

確認できた内容:

- VMD にはセルフシャドウキーフレームを保存する領域がある
- MMD 系資料では「ほとんど使われない機能」とされる

確認ソース:

- `VMDとVPDの概要`

補足:

- UI 上の具体的な編集フローまでは今回の調査では十分に追えていない
- ただし、VMD の保存対象に含まれるため、「本家MMDがセルフシャドウの時系列状態を持てる」こと自体は確認できる

### 1.7 アクセサリキーフレーム

確認できた内容:

- アクセサリ操作パネルから位置、回転、サイズ、表示、影、帰属先を設定し `登録` できる
- フレーム操作パネルの縦軸にも `アクセサリ` がある
- アクセサリを読み込んだあとに `登録` しないままフレーム操作しないよう注意がある

確認ソース:

- `パネル・ボタン一覧`
- `アクセサリを読み込んでみよう`

補足:

- アクセサリキーフレームは PMM の編集機能としては明確に存在する
- 一方で、一般的な VMD の主要カテゴリには含まれていないため、モーションファイル互換という意味では別扱いで考えるのがよい

確認できた内容:

- `空フレーム挿入（ボーンのみ）`
- `列フレーム削除（ボーンのみ）`

確認ソース:

- `メニュー一覧`

補足:

- MMD は時間軸の「間にフレームを差し込む」「列単位で詰める」操作を標準で持つ
- ただし確認できた資料上では対象は `ボーンのみ`

### 2.5 不要フレーム削除

確認できた内容:

- `不要フレーム削除` が標準メニューにある

確認ソース:

- `メニュー一覧`

補足:

- 明示的な仕様説明までは追えていないが、名称からして「冗長なキーの整理」用途の一括処理と考えてよい

## 3. 補間関連機能

### 3.1 補間グラフの直接編集

確認できた内容:

- 補間パネルに `補間グラフ`
- チャンネル選択として `回転・XYZ指定`
- `コピー`, `ペースト`, `線形補間`

確認ソース:

- `基本画面を確認しよう`
- `パネル・ボタン一覧`

補足:

- ボーンの補間は少なくとも `回転 + X/Y/Z` 単位で編集できる
- VMD 上ではボーンとカメラの双方がベジェ補間を保持している

### 3.2 補間自動設定

確認できた内容:

- 補間曲線には `自動設定` 機能がある
- ON の状態でキー登録すると、補間曲線が自動調整される
- 既存キーの間にキーを追加したり削除したりした際、後続キーの補間も再調整される
- VPVP wiki の更新履歴では、Ver.9.23 時点で `自動設定` の ON/OFF ショートカット追加が確認できる

確認ソース:

- `補間曲線操作パネルの「自動設定」`
- VPVP wiki `MikuMikuDance/Ver.7.30以降`

補足:

- 解説記事ベースでは、自動設定の実効対象は主に `センター` と `IK` とされている
- ここは公式仕様というより実動作観察に近いため、「そう見える/そう説明されている機能」として扱うのが安全

## 4. キーフレーム補助機能

### 4.1 バイアス付加・補正系

フレーム編集メニューで確認できた項目:

- カメラ距離バイアス付加
- 視野角バイアス付加
- センター位置バイアス付加
- フレーム位置角度補正
- 表情大きさ補正

確認ソース:

- `メニュー一覧`

補足:

- これらは単純なキー登録だけでなく、既存キーフレーム群に対する後処理機能が本家MMDに存在することを示す
- `MMD_modoki` の比較対象として見るなら、標準 MMD には「タイムライン上の編集補助コマンド」が意外に多い

## 5. 仕様上の注意点

### 5.1 カメラは 1 台、照明は 1 系統

確認できた内容:

- MMD はシーン内に 1 つのカメラしか持たない
- MMD は 1 つの directional light 前提

意味:

- 複数カメラトラックの切替というより、単一カメラのキーフレーム列として扱う
- ライトも同様に単一トラック前提

### 5.2 180 度超の回転の扱い

確認できた内容:

- モデルのボーン回転は、補間時に最短経路側へ倒れるため、180 度を超える動作では中間キーを増やす必要がある
- カメラ回転は、開始と終了の 2 点だけでも多回転を記録できる

確認ソース:

- `アドバイス/カメラ操作について`

意味:

- `MMD_modoki` でボーン編集を本家互換に寄せるなら、回転補間の扱いは重要
- カメラとボーンで回転の体感仕様が異なる

### 5.3 VPD は「1フレームのポーズ」に近い

確認できた内容:

- VPD はボーンとモーフの現在状態を保存する形式
- カメラ、ライト、セルフシャドウ、プロパティのキーフレームは含まない
- 本家MMDは VPD 内のモーフ拡張を標準処理しない

確認ソース:

- `VMDとVPDの概要`

意味:

- VPD は「タイムライン付きアニメーション」ではなく、「単一姿勢の保存/復元」寄り

## 7. MMD_modoki への示唆

本家MMDとの比較で優先度が高そうな要素:

- ボーン、モーフ、カメラ、表示/IK のキーフレーム互換
- 補間グラフ編集
- コピー / ペースト / 反転ペースト
- 範囲選択と時間方向の編集

後回しでもよさそうな要素:

- ライトキーフレーム
- セルフシャドウキーフレーム
- バイアス付加や補正系の一括コマンド
- アクセサリの高度な時系列編集

これは「本家MMDに存在しない」という意味ではなく、`MMD_modoki` の実装優先度としては後段でも成立しやすい、という意味です。
