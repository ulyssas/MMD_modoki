## 1. 目的

設定画面の役割と収録対象を先に整理しておく。

このプロジェクトでは、作業中に頻繁に触る操作と、既定値や実験機能の切り替えを分けて考える必要がある。

整理方針:

- 設定画面 = 既定値と方針を置く場所
- 左パネル / 下パネル / viewport 直下 = 作業中に頻繁に触る操作を置く場所

## 2. 設定画面を増やす理由

現状は、

- UI が各パネルに分散している
- 実験機能の逃がし先が弱い
- 将来入れたいショートカット設定や操作設定の置き場がない

という状態。

特にこのプロジェクトでは、実験機能を本筋の MMD 編集導線から分離したいので、設定画面は意味がある。

加えて、`MMD_modoki` 特有の機能や個人設定をまとめる受け皿としても使いやすい。

## 3. 設定画面に入れるべきもの

### 3-1. 操作設定

- ショートカット設定
- gizmo の既定モード
- local / global の既定値
- Auto Key の既定値
- カメラ操作の既定挙動
- ゲームコントローラ入力マップ
- MIDI キーボード入力マップ

### 3-2. タイムライン / 編集設定

- 波形表示の ON / OFF
- 再生ヘッド追従
- Start / Stop 共通範囲の既定動作
- タイムライン表示密度
- キー表示補助

### 3-3. 表示 / UI 設定

- 言語
- UI 密度
- ラベル表示の簡略化
- 各パネルの既定表示状態

補足:

- 言語設定は上部の常設 UI から外し、設定画面へ寄せてよい
- 常時切り替える操作ではないため、保存される個人設定として扱う方が自然

### 3-4. 描画 / 実行設定

- renderer backend 既定値
- WebGPU / WebGL2 関連の切り替え
- 重いポストエフェクトの既定値
- デバッグ描画の既定値

### 3-5. 出力設定

- 既定解像度
- 既定 FPS
- 既定 codec
- PNG / WebM の既定設定

### 3-6. 実験機能

- 実験機能フラグ
- まだ安定していない描画機能
- 検証用の一時トグル

ここは特に `MMD_modoki` 固有の要素を寄せやすい。

### 3-7. ログ / 診断

- ログレベル
- 診断出力の有無
- 入力イベント記録

## 4. 設定画面に入れない方がよいもの

頻繁に触る操作は設定画面へ入れない。

対象:

- 登録
- 削除
- Start / Stop
- Loop
- gizmo モード切替
- local / global のその場切替
- 現在フレーム移動

これらは作業中に触るので、各操作面に置くべき。

## 5. 想定カテゴリ

最初は次の 5 区分くらいが妥当。

1. 一般
2. 操作
3. タイムライン
4. 描画 / 出力
5. 実験 / 診断

この程度なら過剰に肥大化しにくい。

## 6. 画面形式

設定画面は、常設パネルではなくポップアップ式が本命。

理由:

- 普段の作業画面に常駐させる必要がない
- 左 / 下 / viewport 直下の作業 UI と混ざりにくい
- `MMD_modoki` 特有の機能や実験機能を隔離しやすい
- 個人設定をまとめて見せやすい

そのため、設定画面は「作業中に常時見えているUI」ではなく、「必要なときに開く別面」として考えるのがよい。

導線の本命は、上パネルの歯車アイコンから開く形。

想定:

- 上パネルに歯車マークを置く
- クリックで設定ポップアップを開く
- そこで `言語 / ショートカット / 入力設定 / 実験機能 / 既定値` をまとめて扱う

この導線なら、作業 UI を圧迫せず、`MMD_modoki` 固有機能の入口としても分かりやすい。

## 7. 初期実装の本命

最初から何でも入れない。

初期実装で価値が高い候補:

- ショートカット設定
- ゲームコントローラ / MIDI 入力マップ設定
- 波形表示の ON / OFF
- Auto Key の既定値
- 実験機能トグル
- renderer backend 既定値
- 言語

この辺は「毎回触るわけではないが、保存されてほしい」設定として相性が良い。

## 8. 左 / 下 / viewport 直下との関係

### 左パネル

- 時間軸操作
- Start / Stop
- Loop
- タイムライン

### 下パネル

- キー登録
- キー削除
- 補間
- 値編集

### viewport 直下

- gizmo 連動の現在値操作
- local / global
- gizmo モード
- 必要なら軽い登録導線

### 設定画面

- 既定値
- 機能フラグ
- 保存される個人設定
- `MMD_modoki` 特有の機能
- 上パネルの歯車アイコンから開くポップアップ

## 9. ショートカット設定との関係

ショートカットは設定画面の本命候補。

理由:

- 今後、キー登録系 / 時間操作系 / gizmo 操作系を増やしたい
- 後から固定キーで詰めると衝突しやすい
- MMD 寄りショートカットと一般的な DCC / 動画編集寄りショートカットで好みが分かれる

そのため、少なくとも将来は `ショートカット設定の保存 / 読み込み` を見据えるべき。

## 10. 入力デバイス設定との関係

ゲームコントローラ対応や MIDI キーボード対応も、設定画面と相性が良い。

理由:

- 利用者ごとに必要な入力デバイスが違う
- デフォルトマップを固定すると衝突しやすい
- キー登録、カメラ操作、モーフ操作、再生操作で割り当て先が変わる

設定画面に置きたいもの:

- デバイスごとの有効 / 無効
- 入力マッピング
- 感度や反転などの補助設定
- プロファイル保存

特に `ショートカット / Gamepad / MIDI` は、別々ではなく「入力設定」として同じ画面群で整理した方が見通しが良い。

## 11. 実装順メモ

### Step 1

- 設定画面に入れる候補の棚卸し
- 既定値とライブ操作の切り分け

### Step 2

- 最小カテゴリで設定画面を追加
- 言語 / ショートカット / 実験機能あたりから始める
- その次に Gamepad / MIDI の入力設定を載せる

### Step 3

- 保存 / 復元を実装
- project 保存とは別の、ユーザー設定として扱う

### Step 4

- 操作設定、タイムライン設定、描画設定を段階的に拡張

## 12. 保留事項

- 設定保存先をどこに置くか
- 実験機能をどこまで設定画面に逃がすか
- project ごとの設定と、ユーザー全体設定をどう分けるか
- ショートカット設定 UI をどの粒度で作るか
- Gamepad / MIDI をどこまで共通 UI で扱うか

## 13. 現時点の結論

設定画面は増やしてよいが、作業中に頻繁に触る操作を押し込む場所にはしない。

まずは、

- 言語
- ショートカット
- Gamepad / MIDI 入力設定
- 実験機能
- 保存される既定値

の置き場として整理するのが妥当。

形式としては、常設パネルではなくポップアップ式を本命とする。
