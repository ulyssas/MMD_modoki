# 選択ボーン回転実測値オーバーレイ案メモ

更新日: 2026-04-20

対象:
- `src/timeline.ts`
- `src/ui-controller.ts`
- 必要なら `index.html`

## 1. 目的

次に取りかかる予定の

- 補間曲線編集
- キー編集

を進めやすくするため、タイムライン上で「今の補間結果として回転がどう変化しているか」を見えるようにする。

補間曲線そのものの編集 UI は下パネル補間欄にすでにあるため、タイムライン側では「曲線そのもの」ではなく「結果としての値」を見る役割に寄せる。

## 2. 基本方針

### 2-1. 表示対象は選択中ボーンだけ

常時全ボーンを描くのではなく、選択中ボーンだけを対象にする。

理由:

- 計算量を抑えやすい
- 表示密度が上がりすぎない
- 編集中の対象と表示内容が一致して理解しやすい

### 2-2. タイムライン上では回転の実測値を見る

タイムライン上に出すのは、選択ボーンの回転値推移とする。

- キー位置: そのキーに入っている回転値
- キー間: 補間後に実際に評価された回転値

ここでの「実測値」は、タイムライン表示用にサンプルした補間後の回転値を指す。

### 2-3. 補間曲線そのものは下パネルで編集する

役割分担:

- 下パネル補間欄 = 補間曲線を編集する場所
- タイムライン上オーバーレイ = 補間結果の回転推移を見る場所

同じ情報を二重に見せるのではなく、編集面と結果確認面を分ける。

## 3. 表示イメージ

### 3-1. 基本形

選択中トラックの背面または近傍に、回転値の折れ線または細い連続線を重ねる。

候補:

- `X / Y / Z` を 3 本の細線で同時表示
- `X / Y / Z` を切替表示

初期案としては、まず `X / Y / Z` 同時表示よりも、見やすさ優先で 1 軸表示または簡易 3 軸表示から始めるほうがよい。

### 3-2. キー表示との関係

キーダイヤはそのまま残す。

その上で:

- キー位置では回転値サンプルがキー値と一致する
- キー間では補間の効いた値が線としてつながる

これにより「キーはここにある」「でも実際の変化はこうなっている」が一目で分かる状態を狙う。

### 3-3. `X / Y / Z` の色は既存 UI とそろえる

軸色は新規に決めず、既存の

- gizmo の軸色
- 下パネル補間欄の曲線色

とそろえる。

理由:

- `X / Y / Z` の認識を UI 全体で統一できる
- 学習コストを増やさない
- 補間欄で見ていた軸と、タイムライン上で見ている軸を即座に対応づけられる

表示ルールは、今後ほかのオーバーレイや補助グラフを足す場合にも共通の軸色定義として使い回せるようにしたい。

### 3-4. 選択中トラックだけ縦幅を広げる

回転オーバーレイ表示時は、選択中トラックだけ縦幅を広げる前提で考える。

方針:

- 非選択トラックは現状の行高を維持
- 選択トラックだけ拡張行として表示
- ラベル列も同じ行だけ同じ高さに追従

理由:

- 通常時のタイムライン密度を壊しにくい
- オーバーレイを読めるだけの縦空間を確保できる
- 「どの行の補助表示か」が視覚的に明確になる

この機能では、固定の別帯を増やすよりも「選択行の詳細表示」として扱うほうを本命とする。

## 4. 回転値の扱い

### 4-1. `180 / -180` またぎは飛んでよい

`180 / -180` 境界で線が飛ぶこと自体は、表示として許容する。

今回は無理に unwrap せず、回転値の描画として自然に不連続が見えるほうを優先する。

理由:

- 実際に表示している値の不連続を隠さない
- 表示用の補正ロジックを増やしすぎない
- 「見た目はきれいだが値としては別物」になることを避ける

### 4-2. 値の基準

まずは UI 表示で使っているオイラー角ベースの回転値を前提に考える。

ただし実装時には、どの評価経路から値を取るかを揃える必要がある。

候補:

- 既存キーフレーム値から直接読む
- 補間評価済みの pose をフレームごとにサンプルする

本命は後者。目的が「補間の効き具合を見ること」だからである。

## 5. 実装の切り方

### Step 1

- 選択中ボーン限定
- 回転 `X / Y / Z` のうち 1 軸または簡易 3 軸を表示
- キーフレーム間を一定間隔でサンプルして線を描く

### Step 2

- 軸ごとの色分け整理
- 表示 ON/OFF または軸切替
- サンプル密度の最適化

### Step 3

- 必要ならカメラ回転や他トラックへの横展開を検討

## 6. 非目標

当面は以下を対象にしない。

- 全ボーン同時表示
- 補間曲線そのものの再表示
- 高機能な解析グラフ化
- 複数選択ボーンの重ね描き

## 7. 想定メリット

- 補間をいじった結果が viewport だけに依存せず読める
- キーを足すか、補間を直すかの判断がしやすくなる
- 補間曲線編集とキー編集の前段として、確認コストを下げられる

## 8. 懸念点

- タイムライン表示密度が上がり、キー表示と干渉する可能性がある
- サンプル密度を上げすぎると描画負荷が増える
- 値取得経路が runtime とずれると、見えている線が信用できなくなる

## 9. 現時点の判断

この機能は「左パネルの続き」というより、

- 補間曲線編集支援
- キー編集支援

のための補助表示として扱う。

実装するなら、まずは小さく

- 選択中ボーンだけ
- 回転結果値だけ
- 不連続補正なし

で始めるのが妥当。
