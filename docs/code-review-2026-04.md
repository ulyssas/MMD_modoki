# MMD_modoki コードレビュー (2026-04)

実験的 MMD エディタ・ビューアプロジェクト `MMD_modoki` に対する客観的コードレビューです。

---

## 1. renderer.ts (352行)

**強み:**
- エントリーポイントが明確で、初期化フローが整理されている
- i18n システムが適切に統合されており、多言語対応の基盤がある
- PNG シーケンスと WebM エクスポートが別モードとして分離されており、メインアプリケーションと衝突しない設計
- `installShaderRequestTrace()` による WebGPU/WGSL シェーダーの診断機能が実装済み

**改善点:**
- `initializeApp()` 内で複数の try-catch が存在せず、エラーハンドリングが一箇所に集約されている（良い設計）
- `initializePngSequenceExporter` と `initializeWebmExporter` に重複コードが多い（280行以上の類似処理）
- setStatus, closeExporterWindowSoon などのヘルパー関数が各エクスポート関数内で重複定義されている
- グローバル変数 `shaderRequestTraceInstalled` がモジュールレベルに存在し、状態管理がやや脆弱

**リスク:**
- エクスポート処理のバグ修正時に複数箇所を直す必要があり、メンテナンス性に課題がある

## 2. timeline.ts (653行)

**強み:**
- パフォーマンス最適化が徹底されており、static/overlay/label の3レイヤー構造で最小限の再描画を実現
- RAF (requestAnimationFrame) スケジューラを利用した効率的な描画管理
- バイナリサーチによる可視範囲内キーフレーム検索で大量データでも高速動作
- 垂直方向のカリング処理により、スクロール時も描画負荷が低い
- 双方向スクロール同期が guard フラグで適切に保護されている

**改善点:**
- Diamond マーカー描画が毎フレーム実行される可能性があるが、描画コストは低い
- グローバル定数（RULER_H, ROW_H など）がクラス外に定義されており、カプセル化が不十分
- `selectTrackFromStaticEvent` と `selectTrackFromLabelEvent` にロジック重複がある

**所感:**
- このファイルは非常に高品質で、パフォーマンスを意識した実装がされている。実験的プロジェクトとは思えない完成度。

## 3. main.ts (1068行)

**強み:**
- Electron の IPC ハンドリングが体系的に整理されている
- ファイル I/O、ダイアログ、ウィンドウ管理が機能別に分離されている
- PNG/WebM エクスポートジョブの管理が Map ベースで実装され、複数ジョブの並列実行に対応
- セキュリティ設定が `configureSessionSecurity()` で明示的に管理されている
- `findNearbyFileSync` による柔軟なテクスチャ検索機能が実装済み

**改善点:**
- ファイルが非常に長く（1068行）、複数の責務が混在している（ウィンドウ管理、ファイル I/O、エクスポート管理）
- `sanitizePngSequenceExportRequest` と `sanitizeWebmExportRequest` に重複ロジックが多い
- グローバル Map 変数が多数存在し、状態管理がやや複雑（`pngSequenceExportJobMap`, `webmExportJobMap` など）
- `retainPngSequenceExportOwner` と `retainWebmExportOwner` が完全に重複している
- ストリーム保存の session 管理 (`webmSaveSessionMap`) がグローバル Map で、並行安全性に懸念

**リスク:**
- 開発環境では `webSecurity: false` が設定されており、本番ビルドでも同じ設定が残る可能性がある
- Linux 環境で `--no-sandbox` が使用されており、セキュリティ上のリスクがある

## 4. ui-controller.ts (7785行)

**強み:**
- 機能が非常に充実しており、MMD 編集に必要な UI 操作がほぼ網羅されている
- イベントリスナーの登録が体系的で、DOM 要素との結びつきが明確
- キーフレーム編集、補間曲線、カメラ操作、物理シミュレーションなど複雑な機能が統合されている
- 多言語対応（i18n）が UI 全体に適用されている

**改善点:**
- **ファイルサイズが極端に大きい（7785行）**。これは単一責任の原則に反している
- 複数のコントローラークラスに分割すべき領域（Timeline Controller, Camera Controller, Shader Controller, Keyframe Controller など）
- 大量の private フィールド（300行以上）があり、クラスの状態管理が複雑
- メソッド名が冗長なものが多い（`setupPngSequenceExportStateBridge` など）
- 巨大な switch 文やネストした条件分岐が散見される

**リスク:**
- コードの可読性とメンテナンス性が著しく低い
- 新機能追加やバグ修正のコストが非常に高い
- テストが困難

**推奨:**
- 最低でも 5-10 の責務別クラスに分割すべき（例: TimelineController, CameraController, KeyframeController, ShaderPanelController, ModelController など）

## 5. mmd-manager-x-extension.ts (1460行)

**強み:**
- `.x` ファイルと `.glb` ファイルのアクセサリ読み込み機能が実装されている
- プロトタイプ拡張パターンで既存の `MmdManager` に機能を追加しており、本体コードを汚染しない設計
- ボーンアタッチメント、トランスフォームキーフレーム、親子関係の管理が体系的
- GLB インポート時のデバッグ機能が充実（ログ、バウンディングボックス表示など）

**改善点:**
- デバッグフラグ（`GLB_DEBUG_FORCE_NEON_MATERIAL`, `GLB_DEBUG_SHOW_BOUNDING_BOX` など）がハードコードされており、実行時に切り替えられない
- `createGlbReplacementMeshes` が非常に複雑で、処理内容が追いにくい
- マテリアル変換ロジック（`convertAccessoryMaterialToStandard`）が長く、読みにくい
- WeakMap/WeakSet の使用は良いが、型安全性がやや弱い（`any` 型が多い）

**所感:**
- 拡張機能としては適切に分離されているが、ファイルサイズが大きいため、さらに分割を検討する価値がある

## 9. 全体的な構造とアーキテクチャ

**強み:**
- Electron + Babylon.js + WebGPU の組み合わせで、先進的な技術スタックが採用されている
- ドキュメント（`docs/`）が充実しており、設計意図や調査結果が記録されている
- 型定義（`types.ts`）がある程度整理されており、TypeScript の恩恵を受けている
- 実験的プロジェクトとしては驚くほど機能が充実している

**改善点:**
- **巨大なファイルが複数存在し、モジュール分割が不十分**（ui-controller.ts: 7785行、mmd-manager.ts: 6554行など）
- クラス設計が単一責任の原則に反している箇所が多い
- グローバル変数や Map による状態管理が多く、複雑性が高い
- `any` 型の使用頻度が高く、型安全性が犠牲になっている箇所が散見される
- テストコードが存在しない（実験的プロジェクトのため許容範囲）

**セキュリティリスク:**
- Electron の `webSecurity: false` 設定
- Linux での `--no-sandbox` フラグ
- ユーザーファイルの直接読み込み（file:// プロトコル）

## 10. 推奨事項

### 短期的改善（実験フェーズで対応可能）:
1. `ui-controller.ts` を複数のコントローラーに分割
2. デバッグフラグを環境変数または設定ファイルで切り替え可能にする
3. 重複コード（エクスポート処理など）をヘルパー関数に抽出
4. `any` 型を減らし、型安全性を向上させる

### 中期的改善（正式版を見据えた場合）:
1. `mmd-manager.ts` のリファクタリング（責務別にクラス分割）
2. グローバル状態管理を Redux や MobX などの状態管理ライブラリに移行
3. ユニットテストの導入（少なくとも重要なビジネスロジックに対して）
4. Electron のセキュリティ設定を強化

### 長期的改善（再設計を検討する場合）:
1. Clean Architecture または Hexagonal Architecture の採用
2. ドメインロジックと UI ロジックの完全分離
3. 型安全性を最優先した設計（`any` の完全排除）
4. E2E テストの導入

## 総評

MMD_modoki は実験的プロジェクトとしては非常に高機能で、技術的に野心的な取り組みがされている。特に Timeline の実装やシェーダープリセットシステムは高品質である。

一方で、コードベースの規模が急速に拡大し、特に `ui-controller.ts` と `mmd-manager.ts` が巨大化している点は深刻な技術的負債となっている。

「実験機能の保存」という位置づけを考慮すると、現状のコードは許容範囲だが、今後も機能追加を続ける場合は、早い段階でのリファクタリングが必要である。
