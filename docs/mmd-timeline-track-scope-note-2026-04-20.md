# MMD タイムライン項目スコープメモ

作成日: 2026-04-20

対象:
- `src/timeline.ts`
- `src/ui-controller.ts`
- 将来的には保存/読込系

## 0. 前提

このメモでは、当面の `MMD_modoki` タイムライン設計を次の前提で考える。

- キー打ちは `30fps` 固定フレームベース
- 編集対象は単一シーン

つまり、

- 可変 FPS タイムライン
- 複数シーン / 複数シーケンス管理

は現設計の前提にしない。

将来拡張の可能性は残すが、今の設計判断はまず

- `30fps 固定`
- `単一シーン`

で割り切る。

## 1. 目的

`MMD_modoki` のタイムラインを今後どう広げるか考えるとき、
「ボーンとモーフだけを並べる前提」で考えると MMD 本来の編集対象を見落としやすい。

MMD では、ボーン以外にもタイムライン上の独立項目として少なくとも以下がある。

- カメラ
- 照明
- セルフ影
- 重力
- アクセサリ

このメモでは、MMD 寄せで考えたときにタイムラインへ出す候補項目を整理する。

## 2. MMD で見えている独立項目

スクリーンショットベースで確認できる代表項目は次のとおり。

- `カメラ`
- `照明`
- `セルフ影`
- `重力`
- アクセサリ名ごとの行

つまり MMD のタイムラインは、

- ボーン / モーフの編集タイムライン

だけではなく、

- シーン全体の状態変化
- 演出系の状態変化
- アクセサリ個別の状態変化

も同じ「フレームで管理する対象」として持っている。

## 3. `MMD_modoki` への示唆

### 3-1. タイムライン項目は今後増える前提で考える

今後のタイムライン設計は、

- ボーン
- モーフ

だけで閉じる前提ではなく、

- カメラ系
- ライト系
- 影系
- 物理/重力系
- アクセサリ系

が増える前提で行構造やカテゴリ構造を見ておく必要がある。

### 3-2. 「非ボーン項目」は一括で雑に扱わない

特に次の項目は性質がかなり違う。

- `照明`
  - 色と方向を持つ
- `セルフ影`
  - モードや距離など、照明ともカメラとも違う
- `重力`
  - 物理に関わるシーン設定寄り
- `アクセサリ`
  - モデルとは別の個別トラック群になりやすい

そのため「その他トラック」として一括で押し込むより、
MMD 上どういう独立対象かを意識した分類が必要。

## 4. 項目ごとの重さ感

### 4-1. 比較的入りやすい

- `カメラ`
  - すでにかなり導線がある
- `アクセサリ`
  - 個別トラックとして増える設計にしやすい

### 4-2. 中量

- `照明`
  - UI は既にあるが、常時値からキーフレーム値へ拡張する必要がある

### 4-3. やや重い

- `セルフ影`
  - 専用パラメータの持ち方を決める必要がある
- `重力`
  - 物理設定とアニメーション設定の境界整理が必要

## 5. 現時点の優先順位感

本命の優先度はまだこちら。

- 補間曲線編集
- キー編集
- タイムライン上の複数選択や移動

そのため、非ボーン項目の拡張は今すぐ全部やる話ではない。

ただし、

- 「MMD のタイムラインはボーン以外もかなり多項目」
- 「今後タイムライン項目は確実に増える」

という前提は先に共有しておいたほうがよい。

## 6. 現時点の判断

- 左パネル再整理の話とは分ける
- タイムラインの将来スコープとして別メモで持つ
- 直近では `カメラ / 照明 / セルフ影 / 重力 / アクセサリ` を独立カテゴリ候補として意識する
- ただし実装優先度は、まず補間曲線編集とキー編集の基盤側

## 7. Babylon-mmd に乗る項目と外れる項目

タイムライン項目の重さは、単に項目数が多いことより

- Babylon-mmd のランタイムが直接扱えるもの
- `MMD_modoki` 側で自前評価が必要なもの

が混在することにある。

### 7-1. runtime-native tracks

比較的 Babylon-mmd の既存ランタイムや導線に乗せやすいもの。

- PMX ボーン
- PMX モーフ
- カメラ

これらは VMD 的な発想と近く、既存の再生系や補間評価を活かしやすい。

### 7-2. modoki-owned tracks

Babylon-mmd の外側で `MMD_modoki` が状態を持ち、
フレーム評価して反映する必要が出やすいもの。

- 照明
- セルフ影
- 重力
- 一部アクセサリ系

この層は、少なくとも

- キーフレーム保持
- 補間
- 再生時評価
- 保存 / 読込
- UI との接続

を `MMD_modoki` 側で決める必要がある。

### 7-3. タイムライン UI と評価責務は分けて考える

UI 上は同じ「タイムライン項目」に見せてもよいが、
内部実装では評価責務を分けるほうが自然。

つまり、

- タイムライン表示は共通
- 再生時に誰が値を評価するかは別

という構造を先に認識しておく。

## 8. 今後の実装整理に必要な視点

今後のタイムライン設計では、最初から二層で考えるほうが安全。

1. タイムラインカテゴリを決める
2. `runtime-native tracks` と `modoki-owned tracks` を分ける
3. `modoki-owned tracks` 用の共通キーフレーム格納形式を考える
4. その上で `照明` のような比較的分かりやすい項目から足す

問題は「Babylon-mmd の外の項目を入れるか」ではなく、
「外の項目を modoki 側の正式なトラック種別として扱うか」である。

## 9. 今後メモ化するとよい単位

必要になったら、次は各項目ごとに分ける。

- 照明キーフレーム対応メモ
- セルフ影キーフレーム対応メモ
- 重力キーフレーム対応メモ
- アクセサリタイムライン項目メモ

1 ドキュメント 1 項目に切ったほうが、後で設計判断を追いやすい。

## 10. Babylon 系の参考点

Babylon 側をざっと見ると、`MMD_modoki` が参考にしやすいのは

- タイムライン UI の見た目そのもの

よりも、

- キーの持ち方
- 複数ターゲットの束ね方
- editor 用データと runtime 評価の分け方

のほうである。

### 10-1. Babylon.js 本体の基本構造

Babylon.js 本体では、

- `IAnimationKey`
  - `frame`
  - `value`
  - 必要なら tangent や interpolation
- `AnimationGroup`
  - 複数の `TargetedAnimation` を束ねる

という形で、まず「キー列」と「複数対象の束ね」を分けている。

この発想は `MMD_modoki` に持ち込みやすい。

つまり、

- 各トラックはまずキー列を持つ
- そのトラック群を、再生単位やカテゴリ単位で束ねる

という二段構造を意識すると整理しやすい。

### 10-2. Babylon.js Editor から参考にする点

Babylon.js Editor の公開ドキュメントを見る限り、
Editor は `animationGroup` をシーン上の一級エンティティとして扱っている。

また、スクリプト側では

- `onStart`
- `onUpdate`

のフックを持っており、毎フレーム評価を editor 外のスクリプト側へ逃がせる構造になっている。

ここから参考にできるのは、

- editor 上の項目管理
- runtime 時の値反映

を無理に同一責務にしないこと。

`MMD_modoki` でも、

- タイムラインはトラックとキーを管理する
- runtime-native な項目は Babylon-mmd 側で評価する
- modoki-owned な項目は `MMD_modoki` 側で毎フレーム評価して scene や UI に反映する

という分離が自然。

### 10-3. UI は MMD 寄せ、内部構造は Babylon 寄せ

Babylon Editor は汎用 3D editor 寄りであり、
MMD の

- ボーン
- モーフ
- カメラ
- 照明
- セルフ影
- 重力
- アクセサリ

という並びをそのままは持っていない。

そのため、真似るなら

- UI 導線や項目名は MMD 寄せ
- データ構造と責務分離は Babylon 寄せ

という切り方が妥当。

### 10-4. 特に重要な示唆

Babylon-mmd README では、

- VMD を Babylon.js の `Animation` コンテナで読み込む
- `Self shadow` は not planned

と明記されている。

ここから分かるのは、

- ボーン / モーフ / カメラは Babylon 系ランタイムに寄せやすい
- セルフ影のような項目は最初から `modoki-owned tracks` として考えたほうが筋がよい

ということ。

### 10-5. 現時点の実務判断

Babylon 系を参考にするなら、次に欲しいのは

- トラック共通のキー表現
- トラックカテゴリごとの値型
- runtime-native / modoki-owned の評価分離

であって、タイムライン UI の表面コピーではない。

この方針であれば、

- MMD らしい見た目
- Babylon らしい内部整理

を両立しやすい。

## 11. セルフ影は追いすぎない

MMD の `セルフ影` 項目は独特で、現時点ではそのまま追いかける優先度は高くない。

少なくとも当面は、

- MMD 互換として `セルフ影` を完全再現すること

よりも、

- `MMD_modoki` が既に持っている照明欄と影項目を、タイムライン上でどう扱うか

を先に整理するほうが筋がよい。

## 12. 照明トラックに影項目を含める案を第一候補にする

`MMD_modoki` では現在、照明欄の中に影まわりの項目も持っている。
そのため、タイムライン設計上の第一候補は

- `照明` と `影` を最初から別トラックに分ける

ではなく、

- まずは `照明トラック` の中に影項目も含める

である。

### 12-1. この案を優先する理由

- 既存 UI がすでに照明欄にまとまっている
- ユーザー認知でも「ライト調整の一部」として理解しやすい
- トラック数を増やしすぎずに済む
- 最小実装しやすい

### 12-2. 将来分離する条件

次のような必要が出たら、`影` を独立トラックへ分けることを検討する。

- 影パラメータだけ別頻度でキーを打ちたい
- 照明色 / 方向と影設定を別々に編集したい
- 影がライト以外の要因も強く持ち始める

### 12-3. 現時点の判断

- `セルフ影` はいったん追わない
- `MMD_modoki` では「照明トラック」の中に影項目を含める案を第一候補にする
- 影の独立トラック化は、必要が出たら後で分離する

## 13. アクセサリは内部的には scene object track 寄りで考える

アクセサリまわりは、MMD 文脈では自然に「アクセサリ項目」として見える一方、
`MMD_modoki` ではかなり独自要素が強い。

理由:

- `.x` 読込自体が独自拡張寄り
- 将来的には `obj` など他形式も入れたい
- そうなると「MMD アクセサリ」だけで閉じた設計は後で窮屈になりやすい

そのため、内部設計としては

- `アクセサリ専用トラック`

よりも、

- `scene object track`

として考えるほうがよい。

### 13-1. この考え方の利点

- `.x` 専用設計に閉じない
- 将来の `obj` や他形式も同じ枠に乗せやすい
- transform / visible / scale など共通チャンネルで整理しやすい
- 「オブジェクトをタイムラインで動かす」という Babylon 寄りの一般機能に落とせる

### 13-2. UI と内部の分け方

現時点では、次の分け方が妥当。

- UI 名称や導線は当面 `アクセサリ`
- 内部のトラック設計は `scene object track`
- `.x` はその最初の対応形式

つまり、MMD らしい見え方は維持しつつ、
内部では Babylon 寄りの一般化を優先する。
