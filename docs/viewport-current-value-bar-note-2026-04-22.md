## 1. 目的

`viewport` 直下に置く UI の役割を、下パネルや左パネルと衝突しない形で整理する。

今回の主題は `キー登録 UI をどこへ置くか` ではなく、`viewport 直下を何のための面として使うか` を明確にすることにある。

あわせて、現状の `ボーン欄` をこの新構成へどう移していくかも整理対象に含める。

## 2. 結論

当面の役割分担は次の形にする。

- 左パネル = 時間軸ナビゲーション
- 下パネル = キー編集の主導線
- viewport 直下 = 現在値操作面

つまり、

- 左で「どこを触るか」を決める
- 下で「何をキー化 / 編集するか」を扱う
- viewport 直下で「いまの値をどう動かすか」を扱う

という分担にする。

補足:

- これは `viewport 直下に UI を追加する` というより、`暫定のボーン欄を分解して再配置する` と捉える方が正確である
- 下パネルにはすでに欄が詰まっており、既存ボーン欄を残したまま追加していくと二重化しやすい
- そのため、`viewport 直下の整理` と `ボーン欄の機能切り替え` は同時に考える前提にする

## 3. 現状のボーン欄の扱い

現状の `MMD_modoki` のボーン欄は、最終形というより暫定の値編集欄として置かれている性格が強い。

含まれている要素は大きく 2 系統に分けられる。

### 3-1. viewport 直下へ移す系統

- 対象ボーン名
- `Pos X/Y/Z`
- `Rot X/Y/Z`
- gizmo と連動する現在値表示
- 必要なら小さい `登録`

### 3-2. 新ボーン欄へ残す系統

- `登録`
- `初期化`
- `物理`
- `外部親登録`
- 将来的な選択補助
- 反転ペーストなどの機能操作

つまり、今のボーン欄は将来的に `現在値欄` ではなく `ボーン機能欄` へ変質させる想定で扱う。

## 4. viewport 直下に置くもの

相性が良いのは、gizmo や viewport 上の見た目と距離が近い情報と操作である。

### 4-1. 常時表示したいもの

- 選択対象の現在値表示
- gizmo モード表示
- 座標系表示

具体例:

- ボーン選択時: `Pos X/Y/Z`, `Rot X/Y/Z`
- カメラ選択時: `Pos`, `Rot`, `Dist`, `FoV`
- `Move / Rotate`
- `Local / Global`

### 4-2. 置いてよい操作

- transform / camera 値の直接入力
- gizmo と連動する小さな数値調整
- 補助的な `登録` ボタン

ここでの `登録` は主導線ではなく、`いまの値を見ながらその場で打つ` ための近接導線として扱う。

## 5. viewport 直下に置かないもの

主導線としては下パネルや新ボーン欄へ寄せる。

- 補間編集
- コピー / ペースト / 線形化
- キー削除
- 範囲選択
- 複数キー操作
- 出力設定
- 本格的な登録ボタン群

理由:

- `現在値を触る UI` と `既存キーを編集する UI` が混ざると責務がぶれる
- viewport 直下は面積が限られ、ボタン群を置くとすぐ重く見える
- 下パネルにすでに補間、コピー系、登録系の導線がある

## 6. 新ボーン欄に寄せる機能

viewport 直下へ移さず、ボーン機能欄として残すべきものを明示する。

### 6-1. 主機能

- `登録`
- `初期化`
- `物理`
- `外部親登録`

### 6-2. 後から足しやすい機能

- `コピー`
- `ペースト`
- `反転ペースト`
- `BOX選択`
- `全て選択`
- `未登録選択`

ここは本家 MMD の `ボーン操作欄` に近い性格を持つが、全部を一度に再現するより、まずは `現在値操作` と `機能操作` を分離することを優先する。

### 6-3. 外部親登録の位置づけ

`外部親登録` は情報表示ではなく、選択中ボーンに対する機能操作である。

そのため、

- `情報欄` に置き続けるのは暫定扱い
- `viewport 直下` に入れるものでもない
- 将来的には `新ボーン欄` に寄せる

という整理にする。

## 7. 最小実装案

最初の実装は、現在値バーとしてかなり小さく始める。

### Step 1

- gizmo 状態表示
  - `Move / Rotate`
  - `Local / Global`
- 選択対象の現在値表示

### Step 2

- ボーン値の直接入力
- カメラ値の直接入力

### Step 3

- 必要なら補助 `登録` ボタンを追加

この順で進めると、`viewport 直下 = 現在値操作面` の性格を保ったまま段階的に足せる。

## 8. 移行順

実装は `新設` より `移行` として扱う。

### Step A

- viewport 直下に現在値バーを新設
- ボーン値表示と gizmo 状態表示を繋ぐ

### Step B

- 既存ボーン欄から値スライダー群を外す
- 残った欄をボーン機能欄として組み直す

### Step C

- `外部親登録` を情報欄からボーン機能欄へ移す
- 物理や初期化などの機能ボタン配置を再整理する

この順にすると、既存 UI を壊しきる前に移行先を成立させられる。

## 9. Auto Key との関係

`Auto Key` は viewport 直下 UI と相性は良いが、最初から一緒に入れない方がよい。

理由:

- 現在値表示 / 入力 UI だけでも役割は成立する
- `Auto Key` を入れると発火条件と対象制御の設計が必要になる
- ボーンとカメラで期待する挙動が違う

当面は次の方針が安全。

- まずは手動登録のまま
- `Auto Key` の置き場だけ viewport 直下に想定しておく
- 実装は後段で別判断にする

## 10. 現時点で入れない方がよいもの

- モーフ用の現在値バー
- 複数ボーン登録補助
- 物理焼き込みキー
- 補間の詳細編集
- 複数キー選択との連動

これらは `現在値操作面` よりも `キー編集面` 側の責務が強いか、別タスクとして重い。

## 11. 実装時の注意

- viewport 直下は `値を見る / 値を触る` に徹する
- 下パネルの補間 / コピー系を置き換えない
- ボーンとカメラを優先し、モーフは後回しにする
- `登録` を置くとしても viewport 直下では補助導線に留める
- `Auto Key` は別ステップで対象と発火条件を限定して導入する
- ボーン欄は将来的に値編集欄ではなく機能欄として整理する
- `外部親登録` は情報欄ではなくボーン機能欄へ寄せる前提で設計する

## 12. いまの判断

次に着手するなら、`viewport 直下に現在値操作バーを作る` のと `ボーン欄を機能欄へ寄せる` のを一体の移行として扱うのが最も自然である。

この段階では、

- 見た目の整理
- 値表示
- 数値直接入力
- ボーン欄から値編集要素を外す前提整理
- `外部親登録` の移設先整理

までを主目標にし、キー編集の主導線は引き続き下パネルに置く。

## 13. 実装推奨順

次アップデートまでの現実的な順序は、`編集機能を急がず、まず読み取り専用バーを成立させる` 形がよい。

### P0: 今回の更新に入れやすい順

1. viewport 直下にバーの枠を出す
2. 選択対象名を表示する
3. `Move / Rotate` と `Local / Global` を表示する
4. ボーン選択時の `Pos X/Y/Z`, `Rot X/Y/Z` を読み取り表示する
5. カメラ選択時の `Pos`, `Rot`, `Dist`, `FoV` を読み取り表示する
6. 未選択時 / 再生中の表示ルールを整理する

ここまでは `見た目 + 既存値の読み取り表示` なので、短期で入れやすい。

### P1: 次段で入れる候補

7. ボーン値の数値直接入力
8. カメラ値の数値直接入力
9. viewport 直下の補助 `登録` ボタン

この段階から `現在値操作面` としての実用性が上がるが、入力確定時の反映経路と gizmo 連動の整理が必要になる。

### P2: 後回しにするもの

10. 既存ボーン欄から値スライダー群を外す
11. ボーン欄をボタン中心の `ボーン機能欄` に再構成する
12. `外部親登録` を情報欄から移す
13. `Auto Key` を入れる

この段階は `バーの追加` ではなく `既存 UI の再編` なので、次アップデート直前には入れない方が安全である。

### 判断基準

- 金曜更新に詰めるなら `P0` までを主目標にする
- `P1` は木曜時点で余力がある場合のみ検討する
- `P2` は次回以降の移行タスクとして扱う

## 14. viewport 上コントローラーの扱い

本家 MMD の viewport 右下には、`global / local` や軸方向を示す小さなコントローラーがある。

ただし `MMD_modoki` では Babylon の gizmo がすでに主導線になっているため、これをそのまま再現すると役割が重複しやすい。

当面の方針は次の通り。

- MMD 本家風の浮遊コントローラーは再現しない
- その意味は viewport 直下バーへ吸収する
- 右下に何か残すとしても、小さい read-only 表示に留める

### 14-1. viewport 直下へ吸収するもの

- `Move / Rotate`
- `Global / Local`
- 選択対象名
- 現在値表示

### 14-2. 右下に残してよいもの

- `GLOBAL` / `LOCAL` の小さい状態表示
- gizmo 状態の確認用バッジ

ここではクリック操作や複合コントローラー化までは行わない。

### 14-3. 後回しにするもの

- 本家 MMD 風の右下コントローラー見た目の再現
- 右下 UI からの `Global / Local` 切替
- 軸方向やモードをまとめた複合操作パッド

金曜更新までのスコープでは、`右下コントローラーの見た目再現` ではなく、`意味の整理と役割の移送` を優先する。

## 15. 今週の更新には入れないもの

2026-04-24 の次回更新には、viewport 直下まわりの本実装は入れない前提で考える。

今週の更新で見送るもの:

- viewport 直下の現在値バー実装
- 既存ボーン欄の再編
- `外部親登録` の移設
- `Auto Key`

理由:

- 今回は補間曲線編集まわりと UI 調整ですでに十分な変更量がある
- viewport 直下は `UI を 1 個足す` ではなく `既存ボーン欄の責務分解` を伴う
- 金曜更新前に無理に入れると、見た目と導線の両方が中途半端になりやすい

## 16. 旧 UI からの移送表

来週以降の作業では、既存要素を次のように移す前提で考える。

### 16-1. viewport 直下へ移す

- 対象ボーン名
- `Pos X/Y/Z`
- `Rot X/Y/Z`
- gizmo 状態表示
- `Move / Rotate`
- `Local / Global`

補足:

- ボーンは `gizmo 基準の現在値` を表示する
- カメラは 1 系統だけなので、表示値の分岐は持たない

### 16-2. 新ボーン欄へ寄せる

- `登録`
- `初期化`
- `物理`
- `外部親登録`

### 16-3. 下パネルに残す

- 補間
- コピー / ペースト / 線形化
- 既存キー編集寄りの操作
