# SSAO 現行仕様（2026-03-07）

最終更新: 2026-03-07

## 目的

現時点の `MMD_modoki` に入っている SSAO 実装について、

- どの経路で動くか
- UI から何が触れるか
- 何を近似しているか
- 既知の見た目上の制約は何か

を、実装ベースで残す。

この文書は「調査メモ」ではなく、現行運用の仕様書として扱う。

関連:
- [SSAO 調査メモ（WebGPU）](./ssao-webgpu-investigation.md)
- [ポストエフェクト拡充バックログ](./post-effects-backlog.md)

## 対象コード

- `src/mmd-manager.ts`
- `src/ui-controller.ts`

## 結論

現行の SSAO は、Babylon 標準 `SSAO2RenderingPipeline` をそのまま主経路として使っているわけではない。

現在の主運用は次の通り。

- WebGPU:
  - 独自の fullscreen postprocess `mmdSimpleSsao` を使う
  - 専用 `DepthRenderer` を別管理する
  - depth 差と depth 由来の疑似法線で AO を作る
- WebGL2 / PrePass が安定に使える環境:
  - Babylon 標準 `SSAO2RenderingPipeline` を試行する
  - 失敗時は独自 `mmdSimpleSsao` にフォールバックする

要するに、**現行の実用パスは独自スクリーンスペース AO** である。

## UI 仕様

カメラ選択時のエフェクト欄で公開している SSAO UI は単一フェーダーのみ。

- 項目名: `SSAO`
- 範囲: `0.0 .. 2.0`
- 表示値: 実数 2 桁

内部では後方互換のため次の値も持っているが、通常 UI からは直接操作しない。

- `postEffectSsaoRadius`
- `postEffectSsaoFadeEnd`
- `postEffectSsaoDebugView`

現行 UI の運用値:

- `SSAO` を触ったときは内部で `Radius = 2.0` を使う
- `FadeEnd = 200m` を使う
- `SSAODbg` は UI 非表示

補足:
- `SSAODist` は精度ではなく「どこまで AO を残すか」の距離である
- 近景品質は主に半径、内部解像度、ブラー設計の影響を受ける

## 影・確認用シェーダとの関係

`toon_debug_white_shadow.wgsl` は

- 基本陰影を `50% gray -> white`
- その後に SSAO を追加乗算

の順で合成する。

つまり狙いは、

- 基本の光と影の位置確認
- その上に閉所陰りとして SSAO を重ねる

であり、SSAO が基本影を置き換えるわけではない。

## 実装構成

### 1. 深度入力

SSAO 用に `ssaoDepthRenderer` を別で持つ。

理由:

- DoF 用 depth と用途を分けるため
- WebGPU で AO 専用の更新・再生成を制御しやすくするため
- resize 時に SSAO 専用の再作成を行うため

### 2. 内部解像度

fullscreen SSAO postprocess は、表示サイズに完全一致ではなく、
**短辺 720 を基準にした固定寄り内部解像度** を使う。

意図:

- ウィンドウサイズ依存を少し減らす
- 高解像度時の線状ノイズと二重線感を緩和する
- 負荷を安定化する

ただし完全固定ではないため、ウィンドウサイズ差は軽減であってゼロではない。

### 3. AO の成分分離

AO は 1 種類の値ではなく、おおまかに次の 2 成分へ分けている。

- `detail AO`
  - 溝、重なり、細かい接触を拾う
  - 現状はかなり弱めてある
- `broad AO`
  - 閉所感、囲まれ感、面の沈みを拾う
  - 現在は見た目優先でこちらを主成分としている

最近の調整方針は、
**線を立てるより、閉所の面陰りを残す**
方向である。

### 4. 疑似法線

法線バッファは別途持たず、
depth の左右上下差分から `pseudo normal` を生成している。

用途:

- blur 時の重み付け
- 輪郭をまたいだにじみの抑制
- 正面向き・シルエット付近の suppress

これは見た目改善効率が高い一方、**本物の法線ではない** ため、
髪先や袖端などの細部では不安定さが残る。

### 5. ブラー

ブラーは単純なガウスではなく、depth / 疑似法線ガイド付きで掛けている。

主に 2 段階ある。

- `broad AO` の面用 blur
- 最終 `aoApplied` の mask 平滑化

ただし線状成分を blur しすぎると「レンズボケ風」「二重線風」に見えるため、
最近は次の方針を取っている。

- `detail` を減らす
- `mask blur` で線を再注入しすぎない
- シルエット近傍では suppress を強める

### 6. 近傍 clamp

単発の黒点やフレーム間スパイクを減らすため、
最終 `aoApplied` に近傍 clamp を入れている。

意図:

- 1px だけ極端に暗い値を周辺レンジへ戻す
- 髪先・指先・フェンスなどの黒チラつきを抑える

これは temporal AA ではないので、完全な時間安定化ではない。

## サンプル数

現行の fullscreen SSAO はすでに軽量実装ではない。

概略:

- 本体方向サンプル: 12 方向
- ring 数: 4
- broad blur: 12 方向 + wide blur
- 最終 mask blur: 12 方向
- 近傍 clamp: 4 方向
- 補助 AO 再評価関数あり

そのため、1px あたりの depth 参照回数はそれなりに多い。
今後の見た目改善は、単純なサンプル数追加よりも
`detail / broad` 比率と blur 設計の方が効きやすい。

## 合成仕様

### toon_debug_white_shadow.wgsl

確認用シェーダでは、SSAO は中立色ベースで重ねる。

狙い:

- 白黒確認の邪魔をしない
- 基本陰影位置と閉所陰りを切り分けて確認しやすくする

### MMD Standard

`MMD Standard` 時は、完全な黒乗算ではなく
**Toon 色寄りの自己乗算系合成** を使う。

ただし重要な注意:

- これは postprocess 合成であり
- Toon テクスチャを fullscreen 側で直接サンプリングしているわけではない

現行は最終色から作った `toonTint` 近似色を使っている。

そのため、

- 肌や髪など明るい材質では自然に見えやすい
- 非常に暗い材質では完全な Toon テクスチャ乗算とは一致しない

この制約は仕様として残る。

## 現状の見た目チューニング方針

現在の優先順位は次の通り。

1. 二重線やボケた輪郭を減らす
2. 閉所の面陰りは残す
3. 黒点のちらつきを抑える
4. ウィンドウサイズ差を軽減する

そのため、`SSAO` としては少しマイルド寄りで、
派手なエッジ強調 AO には寄せていない。

## 既知の制約

### 1. シルエット付近の二重線感

完全には消えていない。

理由:

- depth ベース AO なので輪郭不連続に弱い
- 低解像度 AO を拡大している
- 疑似法線はシルエットで不安定になりやすい

### 2. 黒点のチラつき

かなり減らしたが、細線やサブピクセル形状ではまだ出ることがある。

主な発生箇所:

- 髪先
- 指先
- フェンスなどの高周波形状

### 3. ウィンドウサイズ依存

以前より減っているが、完全一致ではない。

理由:

- AO は screen-space
- 内部解像度は固定寄りだが完全固定ではない

### 4. MMD Standard の色付き AO は近似

Toon テクスチャ直読ではないため、厳密な「Toon 色乗算」ではない。

## 運用メモ

現時点の実用上のおすすめは次の通り。

- `SSAO` は中程度で使う
  - 強すぎると線状アーティファクトが目立ちやすい
- `MMD Standard` では色付き AO を使う
- `toon_debug_white_shadow.wgsl` では中立 AO で確認する

## 今後の改善候補

優先度順:

1. `detail` をさらに弱めて `broad/enclosure` 優先へ寄せる
2. temporal stabilize を追加する
3. 本物の法線バッファを別 RT で持つ
4. MMD Standard 時の Toon 色近似を改善する

特に 3 は見た目改善余地が大きいが、実装コストとメモリコストも上がる。
