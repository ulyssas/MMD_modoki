# WebGPU VP8 Encoder Experiment Note 2026-04-22

## 目的

`MMD_modoki` の WebM 出力では、2026-04-22 時点で `WebGPU copy` 経路まで入れたことで一定の速度改善は得られた。
ただし、最終的な主ボトルネックは `GPU -> CPU readback`、特に `GPUBuffer.mapAsync()` 待ちに残っている。

このメモは、その先の発想として

- `WebGPU copy` を前提に
- できるだけ GPU 上で前処理を済ませ
- 可能なら VP8 の一部または大部分を GPU 側で処理する

という実験案を整理するための構想メモである。

これは 2026-04-22 時点では `実装予定確定` ではなく、将来の研究テーマの切り分けを目的とする。

## 前提

- 用途は `MMD_modoki` のオフライン動画書き出しに限定する
- 汎用動画エンコーダを目指さない
- 対象コンテナは `WebM`
- 対象動画 codec はまず `VP8` のみ考える
- 入力は基本的に `RGBA` の 3D レンダリング結果
- 既存の `WebGPU copy` 実装は、`GPUTexture -> GPUBuffer -> CPU RGBA -> VideoSample` の経路

つまり本テーマは、`WebGPU copy` の後段にある `CPU への引き戻し` をできるだけ減らせないか、という問題設定である。

## 背景

現状の実測では、概ね次の傾向が見えている。

- `encode` 自体は軽い
- `draw` は一定コストだが主犯ではない
- `capture` が最も重い
- `capture` の内訳では `mapAsync` 待ちが支配的

このため、

- `VP8/VP9 の選定`
- `keyFrameInterval`
- 軽い UI 更新最適化

よりも、`GPU -> CPU` を避ける構造変更の方が効果が大きい可能性がある。

## この実験で狙うもの

狙いは 3 段階に分けられる。

1. `RGBA -> VP8 入力向け前処理` を WebGPU へ寄せる
2. `VP8 のブロック処理` の一部を WebGPU へ寄せる
3. 最後の bitstream 組み立てだけを CPU で行う

全部を GPU に載せ切ることが目的ではない。
むしろ、

- 色変換
- ブロック分割
- 係数計算
- 一部の予測

のような `並列度が高い箇所` を GPU に寄せ、

- ヘッダ組み立て
- エントロピー符号化
- WebM mux

のような `逐次性が強い箇所` は CPU に残す、というハイブリッド構成を前提にした方が現実的である。

## なぜ VP8 なのか

候補を絞る理由は次の通り。

- 現在の WebM 出力既定が `VP8` に寄っている
- `VP9` より設計負荷を下げやすい
- まずは `MMD_modoki 専用の最小実験` に向く
- 高度な圧縮効率より、`実装成立性` と `速度` を優先したい

この実験の初期段階では、`圧縮率が多少悪くても動くこと` の方が価値が高い。

## エンコードで負荷が高い箇所

VP8 系の処理は、重さの質が違う層に分かれる。

- 画素処理寄りの前段
  - `RGBA -> YUV` 変換
  - マクロブロック / サブブロック分割
  - 差分計算
  - 量子化前の統計計算
- codec 寄りの後段
  - 動き予測探索
  - 係数の整理
  - エントロピー符号化
  - bitstream 組み立て

前者はデータ並列性が高く、`WebGPU` に載せる意味が大きい。後者は逐次性や可変長処理が強く、`CPU` 側へ残る可能性が高い。

## WebGPU 化しやすい箇所

### 1. 色変換

`RGBA -> YUV` は WebGPU と相性がよい。

- 1 pixel ごとの独立処理に近い
- compute shader で書きやすい
- `RGBA` のまま CPU に戻すより筋がよい

最初の候補は `RGBA -> Y plane + UV plane` の生成。

### 2. マクロブロック分割

VP8 の前段では 16x16 / 4x4 単位の処理が多い。
ブロックごとの並列処理に落としやすいので、ここも GPU 向き。

### 3. 前処理系の統計計算

例えば

- ブロック平均との差分
- 分散や簡易コスト計算
- 量子化前の係数準備

などは GPU 化しやすい。

### 4. intra-only 実験

最小実験では `動き予測なし`、つまり intra-only の VP8 風エンコードをまず作るのが現実的。

これなら

- 前フレーム参照
- モーションベクトル探索
- 参照管理

を初期段階で考えなくてよい。

## GPU 化しにくい箇所

### 1. エントロピー符号化

最終的な bitstream 生成は逐次性が強く、GPU と相性が悪い。
ここは CPU 側で行う前提の方が安全。

### 2. WebM mux

これは完全に CPU 側でよい。
既存の mux 経路を流用する方が妥当。

### 3. 完全な VP8 互換実装

VP8 規格に忠実なエンコーダを一から組むのはかなり重い。
最初から完全互換を狙うと、`実験` ではなく `別プロジェクト` になる。

## WebGPU copy の後段を WebGPU で受けたい理由

`WebGPU copy` の実測で見えたのは、`copy / submit` そのものより `mapAsync` 待ち、つまり `GPU -> CPU` へ引き戻した後の readback が大きな壁になっていることだった。

このため、もし専用 encoder を考えるなら、少なくとも次の前段は `WebGPU` のまま受けたい。

- `RGBA -> YUV`
- ブロック分割
- 量子化前の前処理
- intra-only なら予測候補の評価

ここで一度 `CPU` へ戻すと、`WebGPU copy` を使って得た利点がかなり薄れる。したがって、専用 encoder の価値は「全部を GPU に押し込むこと」ではなく、「重い前段を GPU 側に残し、最後の逐次処理だけを CPU に寄せること」にある。

現実的には、

- 前段の重い画素処理は `WebGPU`
- 後段の符号化整理と `WebM mux` は `CPU`

というハイブリッド構成の方が着手しやすい。

## 期待できる速度改善の目安

2026-04-22 時点の実測では、概ね

- `draw` は十数 ms
- `capture` は九十 ms 前後
- `encode` は一 ms 前後

という傾向で、主犯は `GPU -> CPU readback` を含む `capture` 側にある。

このため、もし

- `RGBA -> YUV`
- ブロック分割
- 残差 / 変換 / 量子化の前段
- 係数列生成手前

までを `WebGPU` で処理し、`CPU` に戻すのをかなり圧縮された中間表現にできれば、速度改善の余地は大きい。

理論上のかなり粗い目安は次の通り。

- 控えめな成功
  - `20〜40%` 短縮
  - 前処理だけ GPU 化したが、まだ大きい中間バッファを CPU に戻す場合
- かなりうまくいった場合
  - `40〜60%` 短縮
  - 重い前段を GPU に残し、CPU に戻す量を大きく減らせた場合
- 理想寄り
  - `60〜75%` 短縮
  - CPU 側は最後の逐次的な符号化整理と mux にほぼ限定される場合

逆に、GPU 化しても最終的にフル解像度に近い大きなデータを `CPU` に戻すなら、改善は `10〜20%` 程度に留まる可能性がある。

要するに、鍵は `GPU 化した処理量` そのものではなく、`CPU に戻すデータ量をどれだけ減らせるか` にある。

## 実験の現実的な段階案

### Phase 0: 構想確認

- `WebGPU copy` 後段が本当に壁かを再確認
- どこまで GPU 側で前処理しても意味があるか整理

### Phase 1: GPU 前処理だけ作る

- `RGBA -> YUV` の compute shader
- 16x16 ブロック分割
- CPU 側へ戻すのは `YUV ブロック列` だけにする

この段階では、まだ encode 自体は CPU 側の既存手段を使ってもよい。

### Phase 2: intra-only 試作

- I-frame 的な単独フレームのみ扱う
- ブロック係数の GPU 計算
- CPU 側で簡易 bitstream 化

ここで `画質 / 速度 / 実装難易度` の感触を掴む。

### Phase 3: 実用性判断

- 既存 `WebGPU copy -> VideoSample` より速いか
- 品質が許容範囲か
- 保守コストが釣り合うか

ここで釣り合わなければ、実験メモで止める判断をしてよい。

## このテーマの利点

- `MMD_modoki` 専用用途なので入力条件をかなり固定できる
- クロスプラットフォーム志向を維持しやすい
- native addon や FFmpeg 同梱を前提にしなくても研究できる
- WebGPU / WGSL の実験としても面白い

## このテーマの重さ

正直かなり重い。

- 動けば大きいが、実装の不確実性も高い
- 途中で `結局 CPU に戻す量が多い` 可能性がある
- VP8 互換や mux まで踏み込むと別件化しやすい

したがって、現実的には

- 金曜アップデート向けの短期タスクではない
- 本筋機能を止めてまで優先するものでもない
- `研究枠 / 実験枠` として別扱いするのが妥当

## 当面の非目標

このメモの時点では、次はやらない。

- いきなり完全な VP8 規格実装を始める
- VP9 まで広げる
- 音声 mux まで自前化する
- 既存 MediaBunny 経路を即削除する

## 当面の結論

`WebGPU で VP8 エンコードを一部自作する` という発想自体は、このプロジェクトでは十分面白い。
ただし、いきなり `GPU 直結エンコーダ完成` を狙うのではなく、

- `RGBA -> YUV`
- ブロック前処理
- intra-only の最小試作

の順で小さく確かめるべきである。

このテーマは、`WebGPU copy` の次の一手として価値はあるが、実装着手時は必ず `実験枠` として隔離する。
