# full_light / full_light_add 顔材質調査メモ

更新日: 2026-03-13

## 目的

`full_light` 系の WGSL プリセットで、PMX の toon フラグや toon テクスチャ有無を無視して「常に光面」として扱いたかった。

特にやりたかったのは以下。

- 照明欄の `光色` を 100% 超にしたときの明るさ加算を、顔材質にも他材質と同じように乗せる
- 顔材質が `影なし` 系フラグや独自条件を持っていても無視する
- toon 色指定のない材質にもライトグレー相当の明面色を与える

## やったこと

### 1. `full_light` プリセット追加

`wgsl/full_light.wgsl` を追加し、built-in プリセット `full_light` として選べるようにした。

狙い:

- PMX toon 条件を無視
- 常時明面化
- `光色` と `光色 > 100%` の加算に追従

### 2. `full_shadow` プリセット追加

比較用として `wgsl/full_shadow.wgsl` も追加した。

### 3. 後段の flat light 加算を toon 条件から外した

`src/mmd-manager.ts` 側で、`toonFlatLightMask` / `toonFlatLightColor` の最終加算が `TOON_TEXTURE_COLOR` 条件に縛られないようにした。

目的:

- external WGSL から non-toon 材質にも最終加算を通せるようにする

### 4. `full_light` の後段加算 mask を固定化

`光色 > 100%` 時の加算を材質条件ではなく `lightBoost > 0` だけで通すようにした。

目的:

- 顔材質だけ加算が落ちる現象を減らす

### 5. 半透明レイヤー向け補正

顔材質は透明レイヤーの重なりが強そうだったため、

- `alpha` が低いほど加算色を強める
- `full_light` / `full_light_add` 適用時に `emissiveColor` も持ち上げる

という補正を入れた。

### 6. `full_light_add` プリセット追加

既存の後段加算を直接使うのをやめて、照明欄の値だけ読んで材質内で独自に加算し直す `full_light_add` を作った。

狙い:

- 既存の toon 後段に依存しない
- 顔材質でも各レイヤーに直接足す
- `specular` ではなく前面ベタ加算に寄せる

## 分かったこと

結論として、**問題の主因は PMX の toon フラグではなく、顔材質のレイヤー構成とテクスチャの描き込み** だった。

### 顔材質が他と違う点

- 顔は `body_skin` だけで終わらず、`face00` `face01` `cheek` `eye_*` など複数材質で構成されることがある
- それらの一部は半透明レイヤーとして順番描画される
- 顔テクスチャ自体に頬・口元・鼻筋などの色差や描き込みが入っている

### なぜ髪や服では効いて見えるのか

- 単層寄り
- テクスチャが比較的フラット
- 明るい色が多く、加算が見えやすい

### なぜ顔では効きが弱く見えるのか

- 各レイヤーで加算しても、その後の alpha ブレンドで見え方が変わる
- 後から別レイヤーが重なる
- 顔テクスチャの描き込みがそのまま残る

結果として、材質 WGSL だけで「髪や服とまったく同じ見え方の光面化」を作るのは難しかった。

## 失敗したアプローチ

### テクスチャ明度を潰す

`full_light` で `baseColor` の明度差を消して色相だけ残す実験をしたが、顔が破綻しやすく悪化したため撤回した。

### 透明レイヤーへの加算強化

`alpha` 補正や `emissiveColor` 補正を入れても、顔だけを他材質と完全に同じ見え方へ揃えるには不足した。

### `full_light_add`

`specular` っぽい見え方を避けるため、ベタ加算寄りに寄せたが、顔レイヤーの合成差までは吸収できなかった。

## 今回の判断

`full_light` / `full_light_add` を使って「顔を含む全材質を完全に同じ光面加算で見せる」試みは、いったん打ち切る。

理由:

- 材質 WGSL 単体では顔の多層透明合成に対して決定打がない
- 調整を続けると顔だけ破綻しやすい
- 髪・服・アクセサリ向けとしては一定の用途があるが、顔まで同じ見え方を保証できない

## `full_shadow` の試行

`full_shadow` でも同じ方向の試行を行った。

狙い:

- PMX toon フラグを無視して、常に影面として扱う
- 影欄の `影色` と `Toon影響度` を反映する
- 顔材質だけ真っ黒になるのを避ける

### 試したこと

#### 1. toon 色無し材質向け fallback を追加

toon 色テクスチャが無い材質でも真っ黒にならないよう、ライトグレー寄りの fallback 影色を入れた。

結果:

- 顔の真っ黒化は軽減した
- ただし `MMD Standard` の標準影色とは見た目がズレた

#### 2. 標準影色の式へ寄せる

`shadowTint` だけでなく `Toon影響度` と `toonRaw` を混ぜた標準式に寄せようとした。

結果:

- `shadowMask=1.0` 固定で「常に影面」化する方向は理屈として正しかった
- ただし `@apply-without-toon` の差し込み位置では、そのまま使えない値があり、WebGPU 側で不安定になった
- コンパイルエラーや画面全体の黒化が出た

#### 3. 余計な暗化要素を外す

`shadowRaw` 混合など、標準影色とズレる原因になっていた要素を外す試行もした。

結果:

- 標準影色とのズレは多少減る方向だった
- それでも「標準影と完全一致しつつ常時 shadow 化」は安定しなかった

### `full_shadow` で分かったこと

- `full_shadow` の「顔だけ真っ黒」は、toon 色無し材質の fallback 不足が原因のひとつ
- しかし fallback を入れると、今度は標準影色からズレやすい
- 標準影式そのものを再利用しようとすると、差し込み位置の制約で扱いにくい

### 現時点の判断

`full_shadow` は「標準影と完全一致する固定影」ではなく、**影色ベースの簡易固定影** と割り切る。

理由:

- `full_light` 以上に、標準式の再利用が差し込み位置の制約を受けやすい
- WebGPU 側で黒画や validation error まで出ると、保守コストに見合わない
- 実用上は「真っ黒にならず、影色ベースで落ちる」だけでも用途はある

## 今後やるなら

候補は以下。

### 1. 顔専用プリセットを別で作る

`full_light` と同じにしようとせず、顔用だけ別の見た目を許容する。

### 2. ポストエフェクト側で合成する

材質単位ではなく、最終画面の後段で加算する。

これは顔レイヤーの合成後に効くため、今回の問題に対しては筋が良い。

### 3. 顔オーバーレイ材質の扱いを個別に調べる

モデルごとに `face00` `face01` `cheek` などの役割が違う可能性があるため、対象モデル依存で調整する。

## 現時点のメモ

- `full_light` / `full_light_add` は「顔以外」の材質向けと割り切るなら使い道がある
- 顔を同じ理屈で制御しようとすると、透明レイヤーと描き込みテクスチャで破綻しやすい
- ここを本気でやるなら、WGSL 材質差し替えより後段合成の方が本命

## 用語メモ

### `shadowMask=1.0 固定`

本来のシェーダーでは「このピクセルが明面か暗面か」を計算して、その結果で影色をどれだけ掛けるかを決めている。

`shadowMask=1.0 固定` は、その判定を無視して「常に影面として扱う」意味。

### `toonRaw`

toon テクスチャから引いた段色のこと。標準影では、影色そのものではなく、この toon 段色も混ざる。

### `shadowTint`

照明欄の `影色` に相当する色。ユーザーが UI で決める影の色味。

### `Toon影響度`

照明欄の `Toon影響度` フェーダーに相当する値。`shadowTint` と `toonRaw` をどれくらい混ぜるかを決める。

### `alpha ブレンド`

半透明材質を重ね描きするときの合成。顔はこのレイヤー構成が強いため、材質ごとの加算や影色調整がそのまま見えないことがある。

### `@apply-without-toon`

外部 WGSL を差し込むときの注記。toon テクスチャ前提の分岐に縛られず差し込むための印だが、そのぶん標準の toon 計算をそのまま再利用しにくい場面もある。
